# Seeing Traces, Metrics & Logs in Action

## Introduction

All the configuration is done. Dependencies added, OTLP endpoints configured, Docker Compose ready. Now let's **start the application** and see observability come to life inside Grafana.

This is where everything clicks — logs, traces, and metrics, all interconnected in one centralized UI.

---

## Starting the Application

When you start the Spring Boot application:

1. **Two Docker containers** are created:
   - `dbservice` — the MySQL database
   - `grafana-lgtm-1` — the LGTM stack exposing ports 3000, 4317, and 4318

2. The application starts and begins **exporting telemetry data** to the LGTM stack automatically.

Generate some traffic — invoke REST APIs from Postman, browse the UI, log in with different roles (admin, employer, job seeker) to create a variety of log entries.

## Exploring Logs in Grafana

### Accessing Grafana

Navigate to:
```
http://localhost:3000
```

### Viewing Logs

Go to **Drill Down → Logs**. You'll see:

- **All logs** generated by your backend application in one place
- **Visual graphs** showing log volume over time — info, warning, error counts
- Filter by **service name** (e.g., `job-portal`) — critical when you have multiple applications sending logs to the same backend

### The Data Source: Loki

Loki is the LGTM component responsible for **aggregating and indexing logs**. When you query logs in Grafana, you're querying Loki.

### Filtering and Searching

- Filter by time range (last 15 minutes, last hour, etc.)
- Filter by service name — essential in microservices where many apps send logs
- Search within log content for specific messages

## Understanding Traces — The Real Power

### From Logs to Traces

Open any log entry in Grafana and you'll see a **trace ID** field. This trace ID was added by the **Micrometer library** for every request processed by your application.

### Clicking a Trace ID

Click on the trace ID and Grafana shows you a **beautiful graph visualization** of that request:

```
Client Request: GET /api/users/profile-picture
    └── Controller layer (2ms)
        └── Service layer (5ms)
            └── Repository layer (15ms)
```

### What You Can See:

- The **complete path** of the request through your application
- **Time spent** at each layer — instantly spot performance bottlenecks
- Which method took the longest

### Logs ↔ Traces Are Interlinked

- From a **trace**, click the logs icon → see all logs for that request
- From a **log**, click the trace ID → see the complete trace

This bidirectional linking is incredibly powerful for debugging.

## The Trace ID and Span ID

### Trace ID

A **unique identifier** for an entire request. Every log statement generated while processing that request shares the same trace ID.

**Who generates it?** The Micrometer library.

### Span ID

A unique identifier for a **segment** of work within a request.

**In a monolith:** The span ID is typically the same across all logs because one thread handles the entire request.

**In microservices:** Each service gets its own span ID, but the trace ID remains the same across all services. This is how you track a request as it hops from service A → B → C.

### Adding Trace/Span IDs to Console Logs

By default, trace and span IDs appear in Grafana but **not in console logs**. Update your logback pattern:

```xml
<pattern>
    %d{yyyy-MM-dd HH:mm:ss} [%thread] %highlight(%-5level) %cyan(%logger{36})
    %magenta(%X{traceId}) %magenta(%X{spanId})
    - %msg%n
</pattern>
```

- `%X{traceId}` — the trace ID variable set by Micrometer
- `%X{spanId}` — the span ID variable set by Micrometer

Now your console logs show trace context:

```
2024-01-15 10:30:45 [http-nio-8080-exec-1] INFO  c.e.j.controller.CompanyController
abc123def456 span789xyz
- Entering method getAllCompanies
```

💡 **Note:** Startup logs will have **empty** trace/span IDs because Micrometer only adds them for application request logs, not framework startup logs.

## Exploring Metrics in Grafana

### Accessing Metrics

Go to **Drill Down → Metrics** in Grafana.

### What You See

Micrometer exports every metric that Actuator publishes. In Grafana, each metric has **auto-generated graphs**:

- CPU usage over time
- Memory consumption
- Request counts and latency
- JVM thread counts

### The Data Source: Prometheus → Mimir

- **Prometheus** acts as the data source interface
- **Mimir** stores the actual metric data

### Why This Beats Manual Actuator Checks

Instead of invoking `/actuator/metrics/cpu.usage` manually, you see **trends over time** in visual graphs. Operations teams can:
- Set up **alerts** when CPU exceeds thresholds
- Build **dashboards** combining multiple metrics
- Compare metrics across time periods

## The Complete Observability Workflow

### Debugging a Production Issue

1. **Start with logs** — filter by service name and time range
2. **Find the relevant log** — e.g., an error message
3. **Click the trace ID** — see the complete request path
4. **Identify the bottleneck** — which layer/method took the most time
5. **Check metrics** — was CPU spiking? Were too many threads busy?

### Everything is Connected

```
Logs (Loki) ←→ Traces (Tempo) ←→ Metrics (Prometheus/Mimir)
                    ↕
              Grafana UI (visualization)
```

## ✅ Key Takeaways

- Grafana UI is available at `localhost:3000` when using the Docker LGTM stack
- **Logs** are stored in Loki and filtered by service name and time
- **Traces** show the complete request path with timing for each layer
- **Trace ID** is common across the entire request; **Span ID** varies per thread/service
- Logs and traces are **interlinked** — click from one to navigate to the other
- Add `%X{traceId}` and `%X{spanId}` to logback patterns to see them in console logs
- **Metrics** are visualized as time-series graphs — far more useful than raw API responses
- In microservices, trace ID stays the same across services; span ID differs per service

---

## ⚠️ Common Mistakes

- Forgetting to generate traffic before checking Grafana — you need data first
- Not filtering by service name in Grafana — you'll see logs from all services
- Expecting trace/span IDs in startup logs — Micrometer only adds them to request-handling logs
- Using `sampling.probability=0.1` during debugging — you'll miss 90% of traces

---

## 💡 Pro Tips

- Copy a trace ID from console logs and paste it directly into Grafana's trace search — instant debugging
- In production, span IDs become essential when the same request crosses microservice boundaries
- Grafana supports **alerting** — configure notifications when error log counts spike or latency increases
