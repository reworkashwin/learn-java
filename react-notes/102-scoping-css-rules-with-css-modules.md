# Scoping CSS Rules with CSS Modules

## Introduction

Vanilla CSS is simple and familiar, but the lack of scoping is a real problem as apps grow. A rule in one file can unexpectedly style elements in a completely different component. **CSS Modules** solve this elegantly: you write plain CSS, but the build tool transforms your class names to be unique per file, effectively scoping styles to specific components. It's the best of both worlds.

---

## The Scoping Problem (Revisited)

In `Header.css`, if you change `header p` to just `p`:

```css
p { text-align: center; }
```

Every paragraph in the entire app becomes centered—including labels in `AuthInputs` that shouldn't be. You're forced to carefully manage selectors, which is extra mental overhead.

Wouldn't it be nice if styles in `Header.css` *only* affected the Header component?

---

## What Are CSS Modules?

CSS Modules is **not** a browser feature. It's a build tool feature. Here's how it works:

1. You rename your CSS file from `Component.css` to `Component.module.css`
2. You import it differently—as a JavaScript object instead of a side effect
3. The build tool (Vite, Webpack, etc.) transforms every class name in that file into a **unique, auto-generated name** 
4. You reference the transformed class names through the imported object

### Step-by-Step Setup

**1. Rename the file:**

```
Header.css → Header.module.css
```

The `.module.css` suffix signals the build tool to activate CSS Modules processing.

**2. Change the import:**

```jsx
// Before (Vanilla CSS)
import './Header.css';

// After (CSS Modules)
import classes from './Header.module.css';
```

The `classes` object will contain your original class names mapped to their generated unique versions.

**3. Change how you assign class names to use element selectors for classes:**

First, convert any element selectors to class selectors in your CSS:

```css
/* Header.module.css */
.paragraph {
  text-align: center;
  color: #a39b9b;
}
```

Then in JSX:

```jsx
// Before
<p className="paragraph">...</p>

// After
<p className={classes.paragraph}>...</p>
```

---

## What Happens Under the Hood

When you inspect the rendered DOM, you'll see something like:

```html
<p class="_paragraph_k3x2j_1">Welcome to our app</p>
```

That cryptic name (`_paragraph_k3x2j_1`) is generated by the build tool. It includes:
- Your original class name (`paragraph`)
- A hash unique to this file
- Extra characters to guarantee uniqueness

In the `<head>` section, the injected CSS uses this same generated name:

```css
._paragraph_k3x2j_1 {
  text-align: center;
  color: #a39b9b;
}
```

Now, if another component adds `className="paragraph"` as a plain string, it won't be affected—because the actual CSS rule targets `_paragraph_k3x2j_1`, not `paragraph`.

---

## Scoping in Action

After converting `Header.css` to `Header.module.css`:

- Adding `className="paragraph"` as a plain string in `AuthInputs` → **no styling applied** (the real class name is the generated one)
- Using `classes.paragraph` in `Header.jsx` → **styles applied correctly**

The styles are effectively scoped to the component that imports the module.

---

## Conditional Styling with CSS Modules

CSS Modules work perfectly with conditional class names:

```jsx
<label className={`${classes.label} ${emailNotValid ? classes.invalid : ''}`}>
  Email
</label>
```

Since `classes.label` and `classes.invalid` are just strings (the generated class names), you can use them in template literals exactly like regular class names.

---

## Advantages of CSS Modules

| Advantage | Description |
|-----------|-------------|
| Decoupled from JSX | CSS lives in separate files |
| Standard CSS syntax | No new syntax to learn |
| Scoped by default | Class names are automatically unique per file |
| Team-friendly | Designers can still work on `.module.css` files |

## Disadvantages of CSS Modules

| Disadvantage | Description |
|-------------|-------------|
| Many small files | Each component gets its own CSS file |
| Only classes are scoped | Element selectors (`p`, `div`) are NOT transformed |
| Requires CSS knowledge | You're still writing regular CSS |

---

## ⚠️ Important Limitation

CSS Modules only scope **class selectors**. Element selectors like `p`, `h1`, `div` are **not** transformed and remain global. Always use class selectors in your `.module.css` files.

---

## ✅ Key Takeaways

- Rename files to `.module.css` and import as a JavaScript object to enable CSS Modules
- Class names are automatically transformed to unique values, preventing style collisions
- Access transformed class names via `classes.yourClassName` in JSX
- Element selectors are NOT scoped—only class selectors get the unique name treatment

## ⚠️ Common Mistakes

- Using element selectors in `.module.css` files and expecting them to be scoped—they won't be
- Forgetting to switch from `import './file.css'` to `import classes from './file.module.css'`
- Referencing class names as strings instead of through the imported object

## 💡 Pro Tips

- Use `classes` or `styles` as the import name—both are common conventions
- CSS Modules support composition with `composes:` for reusing styles across modules
- You can mix CSS Modules and Vanilla CSS in the same project—use modules for component-specific styles and a global CSS file for resets/utilities
