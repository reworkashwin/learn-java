# Module Introduction — Practice Projects

## Introduction

You've absorbed a ton of core React concepts — components, state, events, conditional rendering, lists. But absorbing isn't the same as **knowing**. This module is all about **practice**: building two demo projects from scratch to cement what you've learned and prove to yourself that you truly understand React's fundamentals.

---

## What This Section Covers

### 🧠 What is it?

A practice-focused section with **two demo projects** that are basic yet non-trivial. Each project requires you to apply the core concepts you've learned:

- Splitting an app into **multiple components**
- Building an application **from scratch**
- Working with **state** — managing and using it
- **Outputting lists** and **conditional content**

### ❓ Why do we need it?

Reading and following along is one thing. Building on your own is where real understanding happens. These projects are designed to expose gaps in your knowledge and build confidence.

---

## The Approach

### ⚙️ How it works

For each project:

1. **Introduction** — See what the finished app looks like and understand the requirements
2. **Starting code** — Download the provided scaffold with some JSX and utility code
3. **Tasks** — Clear objectives for what you need to build
4. **Try it yourself first** — This is the most important step!
5. **Solution walkthrough** — Step-by-step solution with explanations of design decisions

### 💡 Insight

The solution walkthrough isn't just "here's the code." It explains **why** certain approaches were chosen — giving you insight into how experienced React developers think about component design and state management.

---

## ✅ Key Takeaways

- **Practice is non-negotiable** — try building each project on your own before looking at the solution
- Both projects cover components, state, styling, lists, and conditional content
- Comparing your solution to the instructor's helps identify different valid approaches
- There's rarely one "right" way to build a React app — understanding tradeoffs matters more

## 💡 Pro Tips

- Set a time limit for your attempt (e.g., 45–60 minutes) — if you're stuck, peek at the solution for hints, then try again
- Don't worry if your solution looks different — focus on whether it **works** and whether the code is **readable**
- Take notes on where you got stuck — those are the areas to revisit
