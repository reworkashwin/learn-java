# Adding a Backend to the React SPA

## Introduction

Everything we've built so far works great — in the browser. But refresh the page, and all your posts vanish. Why? Because React is a **frontend library**. It runs in the browser, and all data lives in memory. To persist data and share it across users, we need a **backend** — a separate server that handles data storage, retrieval, and business logic. This section introduces the concept of connecting a React single-page application (SPA) to a backend API.

---

## Concept 1: What is a Single-Page Application (SPA)?

### 🧠 What is it?

A **Single-Page Application** is a web app that loads a single HTML file and then dynamically updates the page content using JavaScript (React, in our case). There's no full page reload — every interaction modifies the existing DOM.

### ❓ Why do we need it?

SPAs provide a smooth, app-like user experience. Opening a modal, adding a post, filtering data — none of these require downloading a new HTML page. React handles all of it in the browser.

### ⚙️ How it works

- Only **one HTML file** (`index.html`) is served to the browser
- This file includes a `<script>` tag that loads the React application
- React takes over, rendering components and managing the DOM
- If you "View Page Source," you'll see a nearly empty HTML file — all the visible content is generated by JavaScript

```html
<!-- index.html — the only HTML file -->
<body>
  <div id="root"></div>
  <script type="module" src="/src/main.jsx"></script>
</body>
```

### 💡 Insight

When you open a modal or navigate between views in a React SPA, no new page is being loaded. React is simply adding, removing, or modifying DOM elements on the same page. That's why it feels so fast.

---

## Concept 2: Why We Need a Backend

### 🧠 What is it?

A **backend** is a separate application running on a server (not in the browser). It provides API endpoints that the frontend can communicate with via HTTP requests to store, retrieve, update, or delete data.

### ❓ Why do we need it?

React runs in the browser — it can't:
- Persist data across page reloads
- Share data between different users
- Interact with databases directly
- Keep data secure from client-side tampering

A backend solves all of these problems.

### ⚙️ How it works

```
┌──────────────┐         HTTP          ┌──────────────┐
│   Frontend   │  ←─── Request ───→    │   Backend    │
│   (React)    │  ←─── Response ──→    │  (Node/API)  │
│   Browser    │                       │   Server     │
└──────────────┘                       └──────┬───────┘
                                              │
                                       ┌──────┴───────┐
                                       │   Database   │
                                       │  (or file)   │
                                       └──────────────┘
```

- The React app sends HTTP requests to the backend (e.g., to create or fetch posts)
- The backend processes the request, interacts with a database or file system
- The backend sends an HTTP response back to the frontend
- React updates the UI based on the response data

### 💡 Insight

React is *not* a backend library. It doesn't run on servers, handle databases, or process API requests. For backend functionality, you use separate tools — Node.js, Python, Java, Go, etc. Frameworks like Next.js blur this line by letting you write backend code alongside React, but pure React is frontend-only.

## Concept 3: The Demo Backend Setup

### 🧠 What is it?

For this course, a pre-built backend is provided — a simple REST API built with **Node.js and Express.js**. It stores posts in a JSON file and exposes endpoints to create and retrieve posts.

### ❓ Why do we need it?

We need a working backend to practice sending HTTP requests from React. Building a backend from scratch is outside the scope of a React course, so a ready-made one is provided.

### ⚙️ How it works

**Setting up the backend:**
```bash
# Navigate to the backend project directory
cd backend

# Install dependencies
npm install

# Start the backend server
npm start
```

The backend runs on `http://localhost:8080` and provides endpoints like:
- `GET /posts` — retrieve all posts
- `POST /posts` — create a new post

### 💡 Insight

You now need **two servers** running simultaneously:
1. The React dev server (frontend) — typically on port `5173` or `3000`
2. The Node.js backend server — on port `8080`

These simulate what a real production setup looks like: two separate applications communicating over HTTP, often on different domains.

## Concept 4: Frontend-Backend Communication

### 🧠 What is it?

The React frontend communicates with the backend by sending **HTTP requests** (using `fetch` or libraries like Axios) and processing the **HTTP responses**. This is how data moves between the browser and the server.

### ❓ Why do we need it?

Without this communication:
- New posts only exist in browser memory — a refresh loses everything
- Other users can't see posts created by someone else
- There's no data persistence at all

### ⚙️ How it works

The general pattern:
1. **Create a post:** React sends a `POST` request to the backend with the post data → backend stores it
2. **Fetch posts:** React sends a `GET` request to the backend → backend responds with all stored posts → React displays them

```
User types post → React packages data → HTTP POST to backend →
Backend saves to file/DB → Sends success response → React updates UI
```

### 💡 Insight

In a deployed application, the frontend and backend would be on different domains (e.g., `myapp.com` for the frontend and `api.myapp.com` for the backend). During development, we use different ports on `localhost` to simulate this.

## Concept 5: REST API Basics

### 🧠 What is it?

A **REST API** (Representational State Transfer) is a backend architecture style where different URLs (endpoints) represent different resources, and HTTP methods (`GET`, `POST`, `PUT`, `DELETE`) represent different operations on those resources.

### ❓ Why do we need it?

REST is the most common pattern for frontend-backend communication. Understanding it is essential for any React developer.

### ⚙️ How it works

| HTTP Method | Endpoint | Action |
|------------|----------|--------|
| `GET` | `/posts` | Fetch all posts |
| `POST` | `/posts` | Create a new post |
| `GET` | `/posts/:id` | Fetch a single post |
| `PUT` | `/posts/:id` | Update a post |
| `DELETE` | `/posts/:id` | Delete a post |

The backend we're using implements these endpoints, and in the next sections, we'll write React code to interact with them.

### 💡 Insight

The backend doesn't care that the frontend is built with React. It just receives HTTP requests and sends HTTP responses. You could replace React with Vue, Angular, or even a mobile app — the backend API stays the same.

---

## ✅ Key Takeaways

- React is a **frontend library** — it runs in the browser and cannot persist data on its own
- A **backend API** is needed for data persistence, multi-user access, and security
- React SPAs use a single HTML file — all UI updates happen through JavaScript/DOM manipulation
- Frontend and backend communicate via **HTTP requests and responses**
- During development, run two servers: one for the React frontend, one for the backend API
- REST APIs use URLs for resources and HTTP methods for operations

## ⚠️ Common Mistakes

- Thinking React can handle backend tasks like database access — it can't (at least, not pure React)
- Forgetting to start the backend server — the frontend won't be able to fetch or save data
- Confusing the development server (which serves your React app) with a backend server (which handles API requests)
- Not understanding that page reload in an SPA loses all in-memory state

## 💡 Pro Tips

- Keep both servers running during development — frontend dev server AND backend API server
- Use different terminal tabs/windows for each server
- The backend's technology stack doesn't matter to React — it just needs to respond to HTTP requests
- Next.js and Remix are React-based frameworks that let you write backend code alongside frontend code — explore them after mastering React fundamentals
