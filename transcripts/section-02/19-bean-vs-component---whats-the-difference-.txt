So far, we discussed on how to create Spring Beans by using @Bean and @Component annotations. Inside this lecture, let's try to understand the difference between them so that you're going to have good clarity on when to use these annotations. When a developer is using @Bean annotation, one or more instances of the class can be added to the Spring Context. So previously we saw, we created multiple Beans of Vehicle class by writing multiple Java methods. But when a developer is using @Component annotation, only one instance of the class or one Bean of the class can be added to the Spring context. The reason is, we're going to mention this annotation on top of the Java class. whereas we're going to use the @Bean annotation on top of the Java method. Inside a Java class, we can write any number of Java methods. Due to this, always go with the @Bean annotation if you have a requirement to create multiple Beans of same data type. Otherwise, you can go with the stereotype annotation. By using @Bean annotation, we can create an object instance of any type of class, including the classes present inside the libraries like String, Integer, Double. Previously, we saw that we're able to create a Bean of String class, Integer class, Double class, and all these are the classes present inside the JDK library or Java library. Whereas, when you are using @Component annotation, we can only create an object instance or a Bean for the application classes only which are created by the Dev team. The reason is, we can't go into the JDK library or to the library classes and mention @Component annotation. We can only reuse the library classes, but we can't edit them. For that reason, by using @Component annotation, we can't create the Spring Beans of the framework or the library related classes. But for the classes which are created by the developer, we have complete freedom to mention this annotation on top of the class. To create a Bean by using @Bean annotation, the developer need to write more code like separate methods to create a bean instance. Whereas creating a bean instance with @Component annotation is very easy, the developer has to simply mention this annotation on top of the class. Moving on to the next difference, while creating a bean by using @Bean annotation, the developer is going to have full control in creating and configuring the bean. When I say full control, inside the method logic the developer can write any kind of logic to initialize the data inside the object or to perform any validations. So all such logic can be written inside the Java method where we have mentioned @Bean annotation. Whereas while we are using @Component annotation, developer will not have any control in creating and configuring the bean. To some extent, developers can use annotations like @PostConstruct, @PreDestroy annotations to have some control to initialize the data or to execute some logic before Destroying the Bean. But they are not going to give full flexibility like in the scenario of @Bean annotation. In the similar lines, while we are using  @Bean annotation, Spring framework is going to create the bean based upon the instructions or the values provided by the developer inside the Java method. Whereas when someone is trying to create a bean by using @Component, the Spring framework is going to take the charge of creating the bean and post that, developer will have access to the object. In this scenario, the developer can't use new operator and invoke the constructors of the class to create the object. The object will be created by the Spring framework behind the scenes. I hope you are clear with these differences. Now I'm assuming you have good knowledge around @Bean annotation and @Component annotation. Please feel free to leverage any of these options based upon your requirements. But in real applications in majority of the times, developers are going to use stereotype annotations like @Component. You are going to witness the same when we start developing an enterprise application by using Spring Boot and Spring Framework.
