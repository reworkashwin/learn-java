By default, eager initialization is going to be followed. Let's try to understand more details about these eager and lazy by looking at their differences. As I said before, eager initialization is the default behavior inside Spring framework. But this default behavior can be changed by using the @Lazy annotation. We can use this annotation with the stereotype annotations or with @Bean annotation as well. When we are using with Eager, the singleton bean will be created during the startup of the application. Whereas the lazy singleton bean will be created when the app is trying to refer the bean for the first time. So the concept of Eager and Lazy, it is only applicable for the singleton bean scope but not for the prototype scope. When we discuss about prototype scope, you're going to get some clarity on why this easy and lazy are not applicable for it. When a singleton scope has Eager initialization, the server or the application will not start if the bean is not able to create due to any dependent exception. So this is how the behavior is going to be. Whereas with the lazy exception, the application will start successfully but at runtime, whenever that singleton bean is getting referred for the first time, at that point of time, if there is an exception happens due to the dependencies, then your application is going to throw an runtime exception in between while you are executing a business logic. As a developer, you should be fine with these default Eager behavior, but there will be certain scenarios where you can go for the Lazy initialization because going with the Lazy initialization has its own advantages. For example, let's imagine there are some thousands of singleton beans inside your application. If you make all these thousands of singleton beans as eager, which is the default behavior, then your spring context will occupy a lot of memory and with that, your startup of the application also is going to be increased because all these thousands of singleton beans, since they're going to be eagerly instantiated during the startup of the application, definitely you will see some time is getting consumed during the startup. Similarly, as if you mark few of the beans as lazy, then your startup time is going to be improved. But don't try to make all your beans as lazy. By marking all your singleton beans as lazy, though your startup time is going to become better, the runtime execution of your business logics will have performance problem. So this performance problem is negligible for smaller or medium applications. Similarly, if someone handles millions of requests of data each minute, but then they may see some performance problems. Let's imagine you marked all these thousands of singleton beans as Lazy. With these, what is going to happen when the end user is triggering some business logic by clicking a button or by navigating to the next page and such scenarios, your business logic is going to be executed and if you are referring all these lazy beans for the first time as part of that scenario, the spring context, it will take few milliseconds of time to create that singleton instance bean for the first time. Whereas with Eager, you're not going to have that problem because all the beans will be created during the startup itself. That's why as a developer, you need to balance between the eager and lazy singleton beans. Most of the times you can go with the eager one. There are certain scenarios where you can go with the lazy option and achieve some improvement if your application is processing humongous amount of data. So what are those scenarios? Let's try to understand the same. Eager can be followed for all singleton beans, which are required very commonly inside an application. Let's consider a e-commerce application. Inside an e-commerce application, searching for the products, placing an order, canceling an order all these are the most commonly executed use cases or business cases. The singleton beans that are handling these business use cases should be marked as eager. So that the end user, whoever trying to execute that scenario for the very first time is not going to get impacted due to the bean initialization. You can consider creating the lazy beans that are going to be used in a very remote scenario inside an application. If you take the same e-commerce application, there will be some remote scenarios or very rare scenarios like closing the account or adding an address or changing the mobile number. These are all the very rare cases and the end user is not going to perform these rare use cases on day to day basis. So for these cases, you can mark those beans as lazy so that that bean will be created only when necessary. If you try to eagerly create all the rare scenario beans as well, you are unnecessarily going to occupy the spring context memory. So that's why based upon your use case, you can switch between these Eager and Lazy at any point of time if you have a confusion, I would advise go with the default eager initialization.
