Singleton scope is a default scope, that means when you are trying to create a bean inside the Spring Framework, that bean is going to have the Singleton scope behind the scenes. This Singleton scope is inspired from the Singleton design pattern that we have, this pattern ensures only one instance of a class is available per application. In the similar lines, Spring Singleton scope also maintains one instance per bean definition within the Spring context. If multiple bean definitions of the same type are declared, then Spring will manage separate single instances for each unique bean definition rather than enforcing a single instance across the entire application. We know there is an option to define multiple beans of a same data type when we are using @Bean annotation. . Basically, we can define any number of java methods that return same Vehicle object or same class data type. On top of all these methods, if you mention @Bean annotation all of them they are going to return multiple beans of same data type. In that scenario, we're going to have multiple bean definitions. For example, if I have multiple methods like v1, v2, v3, which are going to return three different vehicle beans with the Singleton bean scope behind the scenes, the Spring Framework will make sure that each of these bean definitions, they are going to have a separate single instance bean. So this will be v1 bean, and there'll be v2 bean, and there'll be v3 bean. Whereas if you are using stereotype annotation to create a bean, we will left with an option to define only a single bean definition. We can't create multiple beans of same data type when you are using stereotype annotation. In this kind of scenario, since there is only a single bean definition, there will be a single bean instance maintained by the Spring Framework. Now let's try to understand why Spring team, they made the singleton as a default scope and under which scenarios we should be fine with these default scope. The main reason of making every bean as singleton by default is, to achieve the performance. If you're not making the scope as singleton, and if the framework tried to create a new bean every once someone asked, in enterprise application, it is going to have some performance problems, because every time if someone asks if the bean is getting created from scratch behind the scenes, it is going to take some milliseconds and for the enterprise applications where they have humongous amount of data or traffic, they're going to face some performance problems. This is one main reason. And the other reason on why the singleton scope is made as default is usually we will try to create the beans of the classes, which we want to share across the application. For example, if you take these MyService inside this I have defined a constructor and I can define a hundred different methods with various kinds of business logic. Regardless of where this bean is getting used inside my application, I want all of them to have the same business logic. There is no meaning in providing multiple instances or separate instances throughout the application, if my java class is a stateless class and it only carries the business logic. What do I mean by stateless? If your java class, if it is not holding any data that needs to be different for different different requests or for different users, such classes we can call them as stateless because they're not holding any state or data that is unique to different users or different requests that are being processed by your application. So these are the main reasons on why singleton scope is considered as a default scope. Always, when you are having a doubt on which scope to use I would recommend you to go with the default and don't try to change that behavior. In case if your java class holds some data that needs to be unique for different users or different requests, then you can try to change the scope like prototype. When we are discussing the prototype scope, it is going to make more sense to you.
