The spring framework is going to find the right dependency and inject it automatically into a given bean. So this is going to make our applications loosely coupled and easier to maintain as well. With loosely coupled, we're going to get a lot of benefits as well when we are using this auto wiring. These benefits include reduced boilerplate code, improves code readability, supports loose coupling, and spring is going to manage all the dependencies efficiently without me writing any manual code. Inside spring there are three main ways to perform the auto wiring by using Field Injection, Setter Injection and a Constructor Injection. Out of these three Autowiring approaches, Constructor Injection is the most recommended approach for production code. It is the best practice for you to always perform the Autowiring by using the Constructor Injection. Don't try to use the Field Injection or the Setter Injection if you are writing some code for the production application. So there are certain reasons why developers are not encouraged to use this Field Injection and Setter Injection. Let's try to understand them before we try to close this lecture. One of the primary drawback of Field and Setter Injection is, your unit testing is going to become tougher because you can't mark the dependencies very easily. Let me explain what do I mean here. Inside the console, if you try to understand, the Car bean is getting created first followed by the Engine bean, though car has a dependency on the Engine bean, instead of creating the Engine bean first Spring is creating the Car bean first. The reason is very simple. If you go and try to understand the Auto-wiring that we have done. So here, when we are doing the auto-wiring on top of the field or on top of this setter()method, this setter() method or this Field can only be invoked by the spring framework after creating the bean of this Car class. Unless the object of this car Class is available, the spring framework it cannot invoke the engine field to assign the Engine bean into this class and the same applies for the setter() method as well. We know inside Java, we should be able to invoke the instance fields only by using the object of the class. With these reasons, during your unit testing, if you want to replace these engine dependency with some other bean, you need to first create the bean of this Car class, followed by you need to write the manual code of assigning the dependency to a different bean by either invoking the engine field or by invoking the setter() method. Whereas when we are using the Constructor Injection, it is going to be very easy. I'll explain how Constructor Injection is going to work in the next lecture. Moving on to the next disadvantage, whenever we are performing the Field Injection or Setter Injection, we can't mark this dependency field as final. We're going to get the compilation error because if you mark this field as final, once the object is created, the spring framework it cannot assign the actual dependency engine bean to this field either by directly invoking this field or by using the setter() method. Whenever you're not able to use the final before your fields, there is a disadvantage. The disadvantage is your Car object is not going to be an immutable object. It is a mutable object. Anyone should be able to replace these engine bean with some other bean of Engine either by accidentally or intentionally. So that's a second disadvantage. Coming to the third drawback, whenever you are using Field or Setter Injection, the readability of your code is going to be reduced. Another developer cannot easily identify what are the dependencies for your class without opening your Java class. For example in our scenario, unless someone is opening this Car class, they can't understand that this Car class has autowired dependency on the engine by looking at this Field Injection or Setter Injection, whereas with the Constructor Injection we're not going to have this problem. And the last and final disadvantage is there is, a good chance that you may get null pointer injection when you are using Field Injection or Setter Injection. Let's imagine in this scenario, the car bean is successfully created.
