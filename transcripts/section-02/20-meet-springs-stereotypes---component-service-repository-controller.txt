To understand the explanation that I'm going to do about these annotations, first you need to understand what are the different layers that we are going to have in a typical Java backend development. If you consider this box as a Java backend application or any backend application that is developed based upon Python or any other language, the request is going to come into this backend application from the client application. This client application can be a browser application or a mobile application or an microservice or an other third party service, it can be anything. As soon as the request sent to this backend application, it is going to be handled by the controller layer or the presentation layer. This layer is responsible to receive the request from the client applications. Once it performs all the validations on the request, it is going to call the next layer, which is the service layer this layer is also responsible to return the response to the client application once all the business logic is executed. So this is the first layer. After this controller layer, we're going to have the service or business logic layer. Inside this layer only, we are going to write actual business rules, business validations or any data transformations. Once we executed all our business logic, in case if we want to save our data inside a database or if we want to read the data from a database, this service layer it is going to call the repository layer to fetch or store the data. This layer, it is also going to handle the database transactions. If there is an exception happens, it is going to take care of rollbacking of the data that is partially committed onto the database. The last layer that we have after the service layer is Repository or Data Access Layer. Inside this layer, we're going to write all the database interaction related logic. We're going to write a database CRUD related operations. CRUD means create, read, update, delete. All these operations related logic, we are going to write inside this layer itself. From this layer only, the communication with the database is going to happen by using various libraries and frameworks like Spring Data JPA, JDBC. From this repository layer only, we are going to execute the queries onto the database to fetch the data or to delete the data, update the data or insert the data. So this is how that typical backend application is going to be developed. These are the three mandatory layers for any enterprise backend application. You may have a question, why can't we write all the logic inside a single class ? Let me answer the same. Usually in an enterprise application, your business logic is going to have thousands of lines of code. If you try to write all that logic in a single Java class, maintainability of the code is going to be tough and the reusability of the code also is going to be less. Whereas if you try to separate all the logic into these three layers, like controller, service and repository layers, the maintainability is going to easy and the reusability is also possible. Let's imagine you have a method inside this repository layer, which is going to read the data from a given table. If you want to invoke the same method from other service class, you should be able to easily invoke that without duplicating the code. The same applies for service layer as well. If you have certain business validations, which needs to be performed across multiple scenarios, all those logic can be reused if you write the logic inside the service layer. Using these controller layer only, we are going to build REST APIs that we are going to expose to the client applications. Once we start building an enterprise backend application by using Spring Boot and Spring, we're going to use all these standards. If you take this knowledge, now you should be able to understand the purpose of remaining stereotype annotations. You need to mention @Controller annotation on top of the Java classes, which are being maintained in the controller layer. If you are creating a controller class that is responsible to handle the incoming web request, then it is a recommendation to mention @Controller annotation on top of that Java class. Whereas if your Java class is going to present inside the service layer and it handles a business logic, then it is a recommendation to mention @Service annotation on top of that class. In the similar lines, on top of the Java class where you have written all the database related logic, you need to mention @Repository annotation. In case if you are looking to develop REST APIs, then you need to mention @RestController annotation, which is a simplified version of @Controller annotation that return JSON response from the REST APIs logic. Whereas when we are using @Controller annotation, the response will go in a HTML format. We're going to discuss more details about this @RestController annotation in the coming sections. You can assume all these annotations are the children of these @Component annotation. You can use this @Component annotation if you want to create a generic bean, which is not specific to any layer, if there is a class that is going to be used by all the layers and such scenarios you can happily use the @Component annotation. Otherwise you need to use these stereotype annotations based upon the layer where the java class is present. If you try to use the @Component annotation on top of the business logic layer, you're not going to get any compilation errors or the exception, but still using the @Service annotation is the recommendation because they are going to improve the readability of your code. When someone looks at these annotation, they can easily recognize that class as a service layer class. The same applies for @Repository and the @Controller annotations as well. Regardless of which stereotype annotation that you're trying to use, you need to always make sure to use the @ComponentScan annotation on top of the configuration class to enable the automatic scanning of these annotations inside a given package.
