Whenever we add this dependency into the pom.xml, we need to take care of adding the scope as runtime and optional as true as well. This is the standard way of using the DevTools dependency. For other dependencies like spring boot starter mvc, we have mentioned only the group id and the artifact id. . But for the developer productivity related dependencies like H2, we need to take care of mentioning scope and optional. By mentioning the scope as runtime, we are telling to the framework to consider this dependency only during the runtime, but not the compilation time. During the compilation time, there is no dependency on these DevTools. Only during the runtime of the execution of our code, we need this dependency. If you are mentioning a dependency as runtime, then these dependency details is not going to be added to the classpath of your project because there is no compilation dependency. Now let's try to understand what is the purpose of these optional true. By mentioning these optional true, we are making sure that these dependency is not going to be added as a transitive dependency in other projects, which are using the job portal project as a dependency. I know that statement is a very confusing statement. Let me explain you in detail. Let's imagine some other project, they are using the job portal as a dependency in their project. If you don't mention these optional as true inside the pom.xml of job portal, then whatever project which are using the job portal as a dependency, that project is also going to get DevTools as a dependency. Because there is a concept of transitive dependency in maven. Based upon that, since job portal has a dependency on DevTools, if another project uses job portal as a dependency, the DevTools will be added to the new project as well. But we don't want that to happen because we never know that new project, if they want to use the DevTools or not. That's why we need to make sure to mention optional as true and with that, these DevTools is not going to be added as a transitive dependency to the new project. . If a developer really want to use the DevTools, they are free to add these DevTools in their own respective project. That's the purpose of the scope and optional. I hope it is clear. The beauty of DevTools is, when it is trying to deploy our latest code changes, it is not going to restart the application from scratch. Instead it is going to redeploy the code changes which are changed by the developer. That's why we are able to see this efficiency in the startup time when a developer is starting the application manually for the first time. Obviously, all the libraries and their beans, auto-configurations, everything has to be loaded. That's why it has taken 2.3 seconds. But once the server is started successfully and the developer made some changes, this time Spring DevTools, what it is going to do is, it is going to only reload the beans or auto-configurations related to the code changes that we have done. It is not going to touch any of the framework jars or framework related beans because they are already bootstrapped. So behind the scenes, DevTools, it is going to do this kind of optimizations. As a developer, you will never realize that the application is restarted behind the scenes. You can happily keep testing your code changes without restarting the application.
