So now, before I try to get started with the creation of backend REST API, I want to share a standard that you need to follow while creating the packages inside your Spring Boot backend application. Of course, you can create the packages with any name, but in enterprise applications, they're going to follow certain standards. If you learn those standards at this stage, then it is going to make your life easy. Though Spring Boot does not enforce a package structure, but following a clean architecture practices makes our project scalable, readable, and easy to maintain. The first basic rule that you need to take care while creating a Spring Boot backend application is you need to keep your main class at the root package so that Spring Boot can scan all components automatically. For example, if this is your main package, under this package only, we need to have the Spring Boot main class and under this root package, you can create any number of sub packages. While we are trying to create sub packages, there are two popular approaches. The first one is by layer I'll explain to you what do I mean by layer. This is a traditional architecture that developers usually follow. The other approach is by feature or by domain. This is the latest recommendation for large enterprise applications or for microservices. Let's try to understand more details about these approaches. When I say by layer, we are going to organize our code or our packages based upon their technical roles. Long back, we discussed what are the different layers that we are going to have in a typical backend application. We're going to have a controller layer, service layer, and repository layer. Inside the controller layer, we're going to write all the client-facing logic like REST APIs. Whereas inside the service layer, we're going to write all the actual business logic. And under the repository layer, we're going to write all the logic related to database. Similarly, we can create other packages like entity, exception, config. We're going to talk about this entity package when we start discussing about the Spring DataJPA basically, inside this entity package, we're going to place all the POJO classes that represent our database tables. If you closely look at this package structure, all the sub packages are created based upon the technical role that they are playing. All your RestController classes will go inside this package. Similarly, all database related classes or interfaces will go under this repository. This approach is a traditional approach. And from last two, three decades, developers, they are following this approach. The advantages of this approach is it is easy to understand, good for small to medium projects, and it is a common and beginner friendly approach. But coming to the disadvantages, with this approach, it is hard to scale for the large domains or for the large enterprise applications. What is a domain? Domain is a functionality of your application. Like if you take the bank as a enterprise application, under the bank, we're going to have a lot of domains like accounts, cards, loans, marketing, sales. So each of the business department or business entity, we can consider it as a domain. Since enterprise applications, they're going to have a lot of domains in their business. It is going to be hard to scale this package structure because the business logic is spread across multiple layers. So for a small to medium projects, you can happily use this approach. But for the enterprise applications, we need to create the packages by feature or by domain. At the top, we are going to have the root package. Under this, we're going to create sub packages like user, job. If you take our job portal scenario, we're going to have domains or features like user related, job related, company related. That's why in this approach, we're going to create the sub packages based upon the domain name or feature name. And again, under each of these domain package, we're going to have sub packages based upon the technical roles. For example, controller, service, repository, entity, DTO, and so on. The same kind of package structure will repeat for the job domain as well. There are good advantages when we are going with this feature. One of the major advantages, your project, it is going to have high modularity. That means all the logic related to the user, it is going to stay inside these package only. Similarly, all the job related logic will be present under these package. So this way, you are separating the domain logic across multiple modules or across multiple packages. So with this, it is going to be easy to maintain and scale, easier team collaboration, improves domain driven design. You can have multiple teams working independently across multiple packages. For example, the team that focus on job related logic, they always have to work on these and they don't have to worry about the user package. I hope you're clear with these advantages. Now coming to the disadvantage, this approach requires slightly more initial planning. So you have to think about what are the domains available inside your organization. Accordingly, you need to create them. And once the domain packages are created, again, you need to create the technical sub packages. Regardless of which approach you are trying to use, always you can create additional supporting packages. For example, there'll be a config package where you are going to define all the configurations related to your application in the similar lines common, where we are going to place all reusable helper classes or interfaces. Similarly, security, exception, util. So this way you can create any number of supporting packages while creating the packages please make sure you are following these naming conventions.
