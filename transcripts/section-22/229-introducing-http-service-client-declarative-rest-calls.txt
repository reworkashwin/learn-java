Apart from using RestClient to consume the REST services inside Spring and Spring Boot, there is also a better and modern approach is available. The drawback with the RestClient is, the developer has to write each and every logic like invoking the get() method, uri() method, retrive() method, body() method. So whenever developer decides to invoke a REST API, he or she need to take care of writing all the logic around the consuming of the REST service. But with the modern approach that we are going to discuss inside this lecture, we just have to define how our HTTP request should look like. And with that, the framework is going to take care of generating all the invocation logic at the runtime based upon the inputs that we have provided. This is very similar to Spring Data JPA repositories. If you go to any of the Repository interfaces that we have, nowhere we have written any logic to execute the SQL statements. It is the Spring Data JPA framework that is taking care of executing all the logic behind the scenes. . In the very similar lines, we also have an option around the REST APIs invocation. This option name is Http service client. Let's try to build an example so that it is going to be very clear for you. So this time inside the service layer, I'm going to create a new interface. This time we just have to define abstract methods. That's why I'm creating a interface with the name TodoService. On top of this interface, I'm going to mention @HttpExchange. So this is the annotation that we need to use. To this annotation, I'm going to pass the url details of my REST API. Let me invoke url property and to this I'm passing the path as jsonplaceholder.typecode.com slash todos. Now I have to define various abstract methods representing on how my REST API has to be invoked and on how my REST API request is going to look like. Let me paste few lines of code so that we can discuss all of them one by one. All these are abstract methods. The first method name is findAll(). The method names can be anything. The return type of the method should match with the return type of the data that the REST API is going to return. We know findAll() is going to return list of TodoDto, the same I have mentioned here. On top of this method, I have mentioned the annotation @GetExchange. So this is very similar to @GetMapping. When we are trying to produce REST APIs, we need to use @GetMapping annotation. But since this time, since we are trying to consume the other REST APIs, we need to mention this @GetExchange annotation. To this annotation, we can pass the path as well. Instead of mentioning todos here, I can mention to this annotation as well. But I'm going to maintain the todos on top of the interface itself. With this, now my Spring framework, it knows what HTTP method it has to use when invoking this REST API. And since I'm not passing anything as an input, it is going to consume the REST API without passing any request data. Whatever response that my REST API is providing, the framework will try to understand the JSON structure and convert that into a list of TodoDTO. Coming to the next method, which is findById(), we know when we want to fetch a todo based upon a given ID value, we need to pass the ID. So that ID, I want to pass as a path variable. That's why after the @GetExchange, I have mentioned the path variable, and whoever is going to invoke this method, they are going to pass the ID value as an input. So whatever ID value that they are going to provide, that will be binded to this ID path variable. That's why I need to make sure here to mention this @PathVariable annotation. If you are not mentioning this annotation, then this input parameter is going to be treated as a normal method input parameter. So the binding of this input parameter will not happen to this @PathVariable. The same applies for request params as well. In case if you are looking to send some request params or query params while invoking the REST API, please make sure to accept those details as a method input parameters and you need to use @RequestParam annotation, just like how we are using @PathVariable annotation here. Next, I have created a create abstract method and this method is going to accept the object of TodoDto. Since I want to send this entire object under the request body, I need to use these @RequestBody annotation. . If you want to perform some validations, off course you can mention @Valid annotation as well. Since this REST API has to be invoked by using HTTP POST method, I need to make sure to mention the PostExchange method. The response that we are going to receive will be converted into this TodoDto object. Next I'm using PutExchange method because I'm trying to invoke and update REST API. And to this REST API, I need to pass PathVariable along with the UpdatedDto object. That's why I'm trying to accept both of them as an input parameters for this method. Next towards the end, I have created a abstract method, which is going to help me to delete a todo record on the server side. Since it is a delete operation, I need to use @DeleteExchange annotation. To this annotation, I'm passing the PathVariable as an input. So whoever is invoking this abstract method, they need to make sure to pass the ID as a method input value. . This way you can define any number of REST API related abstract methods and all these methods they need to match with your REST API request format and REST API response format along with the path as well. In case if you are looking to send some request Headers, you just have to use @RequestHeader annotation. . To this @RequestHeader annotation, we can provide a value, something like demo followed by I can accept the header value by using a variable. So whoever is trying to invoke this method, they need to pass the path variable along with the @RequestHeader. The framework will try to send all these header values as well inside the request. So as a developer, we don't have to do anything. We just have to define the contract on how our REST API request is going to look like. So once this interface is created, as a very next step, you need to create a configuration class. So let me create a new package with the name config. Inside this package, I'm going to create a new Java class with the name HttpServiceClientConfig. On top of this class, I'm going to mention @Configuration annotation followed by I need to mention and very important annotation with the name @ImportHttpServices. To this annotation, we can pass the package details where our HTTP service interfaces are defined. Otherwise, we can also pass the list of interface names as well by using the types here. To these types, I can pass the interface that we have created. The interface name is, TodoService.class. Based upon these configurations behind the scenes, the framework is going to create a bean of type TodoService and into this TodoService bean, a bean of type RestClient is going to be added as a dependency. Behind the scenes, this HttpServiceClient approach as well, it is going to use the RestClient, it's just that the developer don't have to write all the logic. The developer can simply define the abstract methods and the underlying framework, it is going to take care of generating the actual logic by using the RestClient. As you can see, the job of you is going to be very easy because the underlying framework is taking care of writing or building the actual REST API invocation logic. You can go with the RestClient approach only in the scenarios where you want to have full control on how you are invoking your REST API. Most of the scenarios should be achievable by using HttpServiceClient approach as well. But in case if there is any peculiar scenario or some rare scenario where you are missing some control on how the REST APIs are getting invoked, then you can happily adopt the RestClient approach.
