We know on how the web applications are going to work, the client applications are going to send the request by using the HTTP protocol. So whatever request that we are getting by using the HTTP protocol from the client applications, in other words, we can also call it as HTTP request. When we want to create a bean by using the @Request Scope, we need to use this annotation and whenever we are using this annotation, the bean will be created for each and every HTTP request that is being processed by an web application. Just like how a prototype scope bean is going to be created whenever someone is trying to refer that bean, in the similar styles, whenever someone is sending requests to the web applications by using the HTTP request, in all such scenarios, a new bean will be created. Whatever bean that get created, it is only valid for the duration of a HTTP request. We need to use these scope when we want to store some request specific data, like Storing temporary user inputs like form data. So this is the syntax that we need to follow to create a bean of request bean scope. If multiple users sends multiple requests, each of them, they're going to get a separate request bean or separate instance of the class. As soon as the request is complete, the bean is going to be destroyed. Next we have session bean scope. Just like how we have HTTP request, we also have a concept called HTTP session. When a user started interacting with a web application from a browser, a new HTTP session will be created and the same session will be maintained until the browser is closed or the session is expired. When we create a bean with the session scope, a new bean instance will be created for each user HTTP session. We need to use these bean scope when we want to store user specific data across multiple requests in a single session. For example, if I want to store some logged in user details or shopping cart information, which needs to be survived across multiple HTTP requests, then this scope is a good consideration. This is the syntax that we need to follow to create a bean with this scope. As I said before, whenever the end user opened a browser and trying to access a web application, at that specific point of time, a session scope to bean is going to be created. The same bean instance will be used across multiple requests from the same user. As soon as the session expires, the bean is going to be destroyed. Next we have Application Bean Scope. When we are defining a bean with this scope, a single bean instance is going to be shared across the entire application. If the application is being used by millions of end users, they may have millions of sessions and millions of HTTP requests sending to the backend for all the sessions and for all the requests and for the entire application. If you want to have a single bean, then you need to define the bean with the application scope. This scope needs to be used when we want to store global data shared across all users and all requests. For example, I can store some global statistics data, like number of visitors that are visiting my website. This is the syntax that we need to follow when we are using this scope, only one instance of the bean is going to be created across entire application. The bean is going to be destroyed only when the application stops. Next we have Web Socket Bean Scope. You will see this scope is being used very rarely by the developers because this scope is limited to only web socket scenario. So what is a web socket? Web socket is a continuous connection between the client and the server. Unlike HTTP, which works using request response, web socket stays open and allows real-time two-way communication. Whenever you are chatting with WhatsApp or Facebook Messenger or any other chat applications, in those scenarios, the web socket protocol is going to be used. Because using HTTP there is going to be a costly process. If you want to create a HTTP request for each and every message that someone is trying to send, it is not going to be a scalable solution. That's why under the scenarios of live chat, stock price updates, notifications, multiple player games, in all these scenarios, web socket protocol is used extensively. Whenever you are building business logic for these kinds of scenarios, you can mark the bean with this scope by using this annotation. With this, what is going to happen? A bean will be created for each web socket connection and destroyed whenever the connection closes. To take an example of chatting application itself, each web socket client who initiated the chat, they're going to get their own bean or object.
