Let me try to answer the same. The main difference between the application scope and singleton scope is, who create and maintains a bean. . We know Application Scope we can only use when we are developing the web application. When we mark a bean with this annotation, the bean instance will be created and managed by the Servlet context. We know all our web applications, they're going to process the request and response by using the servlets behind the scenes. That's why we deploy our web applications into the servlet containers like Tomcat. Inside that servlet container, there is a concept called servlet context. So this servlet context will create and maintain the ApplicationScopeBean. Typically, you're going to have one instance of ApplicationScope bean for your entire Spring web application. But there are some rare scenarios where developers, they will try to deploy multiple web applications into a single Tomcat server or into a single servllet container. So in these kinds of scenarios, each web application, they're going to have their own separate servlet context under the servlet container. Let's imagine under a servlet container or under a Tomcat server, there are two applications deployed like application1 and application2. So both these applications are going to have their own servlet context. In this kind of scenario, if both these application1 and application2 has the same code, the servlet context will create a separate ApplicationScopedBean for both these servlet context. But usually inside a web application, we are going to deploy only a single web application into a single servlet container. That's why most of the times we are going to have only one instance of Applicationscope bean inside the servlet container. Whereas coming to the singleton scope beans, they are going to be created and maintained by the Spring container. When we are discussing about the basics of Spring framework, we created the object of Spring container by using a class called AnnotationConfig Application context. So whenever we are creating a bean of this class, a Spring container will be created behind the scenes. If you are trying to create multiple spring containers inside a spring application, then multiple instances of the singleton bin will be created. But 99% of the times developers, they're going to have only a single Spring container inside a Spring application. That's why most of the times we will see one singleton instance bean available throughout the application. . I hope you understand the primary difference between these scope beans. In the scenario of application scope, the soullet context will create and manage the bean, whereas in the scenario of singleton, the spring container will create and manage the bean. So this application scope, it is going to work only inside of web applications like Spring MVC applications or Spring Boot based web application, whereas singleton scope, it is going to work in all type of applications. It can be a Web CLI or Batch applications as well. To configure these scope, we need to use this annotation on top of the Java class, whereas to use the singleton scope, we don't have to mention any annotation because that is the default scope inside the spring ecosystem. The bean that is created with the application scope, it is going to be shared across the entire application for all users and HTTP requests and HTTP sessions, whereas the bean of singleton will be shared across the entire Spring container. The application scope bean will be destroyed when the application stops, whereas the singleton bean, it is going to be destroyed as soon as the Spring container shutdown. The use cases to use the application scope is, to store global application state like visitor count, application wide configurations and so on, whereas for the singleton, the use cases will be, if you want to have a bean which can be accessed throughout your application, you can mark such classes as singleton. For example, if you want to create a service class or repository class or component class, which can be accessed throughout your application, for all these scenarios, we can go with this singleton default scope, and with that, the same bean instance will be leveraged across our entire business logic. For the 99% of the scenarios, you should be fine with this singleton scope. Don't create this application scope unless you have a scenario to store the global data and share across the request and sessions.
