This is the internal flow of Spring Security Framework there are good amount of individual components which are playing different roles inside the authentication flow. When an end user is trying to access an REST API or a web path, the Spring Security Framework first it will decide whether a given path is a secured path or a public path. If it is a public path, none of this flow is going to come into picture, the end user request will directly send to the controller layer. So after this Spring Security layer, we are going to have the controller layer towards the end. In the case of public APIs or public paths, the request will be directly forwarded to the controller layer. Whereas if the end user is trying to access a secured API path, the Spring Security Framework identifies the same and it will check whether the end user authentication is already completed or it is yet to complete. All these housekeeping activities will be done by the Spring Security filters. Inside the Spring Security Framework, there are good amount of filters like around 15 to 20 filters are there performing various different activities. One of the primary responsibilities of the Spring Security filters is, to understand whether a given request API path is a secured path or a public path. If it is a secured path, they need to identify whether the given end user is already authenticated in the previous sessions or not. If the end user is not previously authenticated and he's trying to access the secured API path, these filters are going to take care of redirecting the end user to the login page. We saw a default login page as well in the previous lectures. Once the end user submitted the credentials, one of the filters present inside this Spring Security filter, it is going to extract the username and password from the HTTP request and convert that into an authentication object. This authentication represents an end user credentials along with the username and password. This authentication is also going to have information to identify whether the authentication is successful or not. Once this authentication object is populated by the Spring Security filters, the request will be forwarded to the authentication manager. So whatever I have highlighted here like Authentication, Authentication Manager, Authentication Provider, Password Encoder, UserDetailsManager, UserDetailsService, all these are interfaces. Behind the scenes, Spring Security Framework, they also provided the implementation classes for these interfaces as well. The implementation class of this AuthenticationManager, it is going to take care of triggering the authentication. So the whole responsibility of performing the authentication is with this AuthenticationManager. To this Authentication Manager, the Spring Security Framework, it is also going to pass the authentication implementation class object. So the Authentication Manager has all the details to initiate the authentication like username and password. This Authentication Manager, it is only going to take care of initiating the authentication, but it is not going to perform the actual authentication. The actual authentication will be taken care of by the Authentication providers. In a Spring Boot application that is using Spring Security Framework, it can have any number of Authentication Providers. That's why I have mentioned multiple boxes here. The reason why Spring Security supports multiple Authentication Providers is, usually in a web application, they're going to support multiple styles of authentication. The end user can trigger the authentication by simply using the OTP. There is also an option of submitting username and password. So there are multiple flavors of authentication,to support all these multiple flavors of authentication, we can define multiple Authentication Providers. The job of this Authentication Manager is to pass the authentication object details to all the Authentication Providers that are registered inside a web application. It is going to iterate all the applicable Authentication Providers, and it will ask the Authentication Providers to perform the authentication. If any of the Authentication Provider confirms that the authentication is successful, the same information will be sent back to the Spring Security filters and Spring Security filters, they are going to store the authentication details into the Security Context so that in future, if the same end user is trying to access different security API from the same browser, the Spring Security filters, they are not going to redirect the end user to the login page. Instead, they're going to leverage the authentication details already stored inside the Security Context, and accordingly, they're going to forward the request to the controller layer. This way Spring Security filters, they're going to avoid triggering the authentication logic for each and every request. We already saw a demo in the previous lectures. By default, Spring Security framework, it is going to create a cookie with the name JSESSIONID. When an end user is trying to make a request for the very first time, he will be displayed a login page. The end user will enter the credentials, the credentials are valid, and if the authentication is successful, those details will be stored inside the security context along with the JSESSIONID. The same JSESSIONID will also be sent inside the response to the browser or to the client applications. The client applications, they're going to store this information inside cookie. In future, when the same end user is trying to access a secured REST API, the browser, it is going to automatically attach the JSESSIONID cookie to the request. If this cookie is present inside the request, the Spring Security filters, they're going to validate if the cookie value that they stored inside the security context is same. If yes, they're going to ignore all this authentication logic and forward the user request directly to the controller layer. So the actual logic of authentication is going to be present inside the Authentication Providers. These Authentication Providers they're going to take help from two different components to perform the authentication. The first component is UserDetailsManager or UserDetailsService. At the end of the day, if the end user is trying to log in, we need to validate whether the login details are matching with what we already stored inside the database. To perform that comparison of the credentials, first, we need to load the existing credentials from the storage system like database. That responsibility of loading the user details from the database or from a storage system will be taken care by these UserDetailsManager or UserDetailsServiceComponent. To this component, we just have to pass the username that we received. Based upon the username, this component will load the user details from the database and the same will be forwarded to these AuthenticationProvider. Inside the Authentication Provider, the password comparisons will happen,we're going to have two different passwords, the password that we received from the end user and the other password that we received from the storage systems like database. Both of them will be compared. If they are same, the authentication will be considered as success otherwise, it will be considered as failure. There is also another component with the name Password Encoder inside the Spring security framework. Passwords are sensitive fields we should never store them inside database or in any other storage systems using plain text. If my password is Madan, the same value should not be stored as it is in plain text format in the storage systems. If someone is storing the passwords using plain text, it is a serious problem because anyone who has access to the database or to the storage system, they should be able to query that table and identify the passwords of all end users. To avoid such problems, we have this Password Encoder interface. There are good amount of implementation classes for this interface. The responsibility of these password encoders is to hash our passwords before storing them into the database. Hashing is a concept where if you provide plain text as an input to these hashing algorithms, they're going to convert that password into a format like encrypted format, which no one can understand. We're going to see the demo of these Password Encoders in the coming lectures. The responsibility of this Password Encoder is, to store the passwords in a properly secured manner. While we are registering a new user, we need to take the help of PasswordEncoder to store the password in a proper encrypted format. If the same user is trying to attempt the login operation, these Authentication Providers, they are going to take help from the Password Encoders to identify if the user entered password and the already stored passwords are same or not. Hashing is a very beautiful concept. I discussed about this hashing in detail inside my Spring Security course. Using hashing, we can convert a plain text value into a hash value. By using this hash value, the reverse mechanism is not possible, which means from this hash value, no one can understand what is the original plain text value. Even if someone tried to manage to hack your database by looking all the hash values that we stored inside the database, they cannot guess or they can't know the plain text passwords.
