First, let's try to understand what are the various components involved when we try to implement observability in Spring Boot application. We can think of observability as a pipeline. At the starting of the pipeline, the Spring Boot application is going to be present. This Spring Boot application is going to generate the logs. It is also going to expose some metrics information by using the actuator. All these telemetry data, has to be transported to a observability backend product like Grafana. . Once that data is stored in a centralized backend location, using the same, we can try to build the dashboards and visualize all the telemetry data. So this is a typical flow that is going to happen when we try to implement observability in a Spring Boot application. At the very first step, we're going to have Spring Boot application. Inside this Spring Boot application, in general, we're going to have business logic that handles requests. This application is going to talk to database, but it does not natively know how to expose its metrics, traces, or logs information in a standardized way. So this is where Spring Boot framework need help from the observability libraries. One such library is Micrometer. Micrometer is Spring's abstraction layer for observability. We can think of it like an SLF4J for metrics and tracing, just like how we have SLF4J inside logging, which is going to act as a abstraction layer. In the similar lines, this micrometer is going to act as a abstraction layer for generating metrics and traces information. Logging is going to be generated by the Spring Boot framework by using the logging framework. So this micrometer is only going to take care of generating metrics and tracing information. How it is going to collect? By deeply integrating with the Spring Boot actuator. We know inside a Spring Boot backend application, there is a component called actuator, which is going to take care of exposing the metrics. These micrometer libraries, they're going to collect the metrics like CPU, memory requests, count, latency, and so on from the Spring Boot actuator. Apart from collecting the metrics, this micrometer, it is also going to add the tracing related information via micrometer tracing. This micrometer, it is only going to take care of generating the metrics and tracing information, but it does not store the data. So micrometer collects the telemetry data and forwards the data. To whom the data will be forwarded? It will be to the open telemetry. Open telemetry in short form, we can call it as OTel. This is a standard observability framework, which is going to define how telemetry data should look like, how it should be collected and how it should be exported. So based upon the standards of these open telemetry, the micrometer, it is going to collect all the telemetry data and forward the same to this framework. That's why in simple words, we can say micrometer will bridge that data to the open telemetry. And this open telemetry, we know internally, it is going to provide various APIs, SDKs, vendor neutral standards. So once that data is received by this OTel framework, its responsibility is to export data into an observability backend product. Inside industry, we have various observability backend products. Think like this is a vendor1, vendor2, and this is vendor3. This way there are multiple vendors. Before open telemetry, each of these vendor, they used to have their own standards to accept the telemetry data, but with the introduction of open telemetry, there is a new protocol is introduced with the name OTLP. By using this protocol, this open telemetry framework, it is going to export all the telemetry data in a vendor neutral standard. OTLP is a wire protocol. It defines how telemetry data has to be sent to the observability backend systems. Today, I might be using a vendor1 product and tomorrow, if I want to switch to vendor2 product, there will not be any code changes in my SpringBoot backend application because this open telemetry is following the OTLP protocol, which is a vendor neutral standard. So this protocol works over HTTP and GRPC, just like how we have REST for HTTP protocol. Similarly, OTLP is for observability. Because of OTLP, SpringBoot does not care where telemetry data is going. And similarly, the observability backend products, they don't care who is producing the data. There is no tight coupling between the SpringBoot backend application and the observability backend applications. Once the data reaches to these open telemetry from the micrometer, this telemetry data will be exported to the observability backend by using these OTLP protocol. Once the data is exported, there has to be a observability backend on the other side to accept all these data that is being exported. In our demos, we are going to use LGTM stack from the Grafana as a backend. Grafana is a famous product, which has various other sub-products supporting observability concepts. So each of these word has a meaning. L means Loki. Using this Loki, we're going to store all our logs in a centralized location so that we can search these logs visually inside Grafana. After L we have G, G means Grafana. Grafana is going to act as a UI. By using this UI, we can visualize the logs, metrics, tracing information we can build the dashboards. Next we have T, T means Tempo. By using this Tempo tool, we're going to store the traces information that we are receiving from the SpringBoot application. So this traces information we can see visually as well inside the Grafana tool. At last we have M, M for Mimir. By using this tool, we can store the metrics, which can be visualized inside Grafana. These tools are going to store telemetry data and give flexibility to us to query and visualize based upon our requirements. They do not collect data directly from our SpringBoot application. Instead, they're going to rely on the middlemans like OpenTelemetry, which are going to expose the telemetry data in a vendor neutral format by using OTLP protocol. This is how we can visualize this observability pipeline. At the very starting of the pipeline, we're going to have a backend application or an infrastructure, which is going to generate telemetry data like logs, metrics, and tracing information. . In the scenario of SpringBoot, it is going to depend on the micrometer library to generate all the telemetry data in a format that these OTel will accept. By using these OTel and OTLP, we are going to collect, manage, and export telemetry data.
