In the scenarios where your request is being processed by multiple threads under such scenarios, the micrometer library, it is going to add a different spanId for each of the thread that is handling your request. But the trace ID will be same for the entire request. Usually the spanId is more valuable or more relevant in the microservices environment because the same request from the client application will travel from microservice A to B and from B to C and all these microservices, since they are going to be deployed in separate servers, definitely the same request will be handled by multiple threads. In such a distributed environment where a single request is being processed by multiple services, we are going to have the same trace ID value, but each of the microservice logs, they're going to have their own respective spanId. But right now our job portal application is a monolithic application, and since we have a single backend application, a same thread is going to handle the entire request in our scenario.
