Before we try to implement Authorization inside our application, first let's try to understand what is the difference between Authentication and Authorization. By using authentication, we are going to identify who is the user that is trying to access and secured system. Whereas by using authorization, we are going to check whether the given authenticated user has enough privileges or authorities or roles to access a secure resource. Always, authentication is going to happen before the authorization. Authentication is also going to be represented in a short form by using AuthN. After the authentication, authorization will come into picture. To perform the authentication, we only require end-user login details like email and password. But to enforce authorization, we require end-user privileges or roles. Usually during the login operation, once the given username and password are correct, we are going to load the corresponding privileges or roles of the user from the database or from any storage system. Once the roles or privileges of the user are loaded, we are going to perform the authorization. If the user has enough privileges or roles, then the authorization is going to pass and the user will be provided an access. In a typical backend application, it is completely possible to build REST APIs that can be invoked by any logged-in user, or we can enforce some extra restrictions as well so that only a logged-in user with so-and-so privileges should be able to access this secure API. We are going to discuss about that in the coming lectures. There are separate status codes for these authentication and authorization. Whenever the authentication itself is failing, as a standard, we need to return 401 error. 401 means the end-user credentials are invalid. Whereas, if the authorization fails, we should return 403 status code. 403 means forbidden error. By using this 403, we are going to tell to the end-user, I accept that you are a valid authenticated user, but you don't have enough privileges to access so-and-so resource. For example, if you take an application where there are two types of users exist, the first one is customer, the other one is employee, both of them, they can perform various kinds of actions inside the application. But to perform those actions, first they need to prove their identity. Once the authentication is completed and logged in into the application based upon their roles and authorities, they can perform various kinds of actions. Let me give a real-life analogy as well to remember these authentication and authorization. Let's imagine you are traveling from New York to Dubai. So to travel from one city to other city, you usually go to an airport. So at the starting of the airport, you need to prove your identity. So the security guy, he's going to ask you to show your passport and ticket details. As long as your passport and ticket details are correct, the security guy, he's never going to worry about which terminal or gate you are going once you enter into the airport. Because the role of the security guy is only to perform the authentication. So how he's performing the authentication by making sure that you have a ticket on the same day and you are the same person who booked the ticket by looking at your passport. After the authentication, you're going to enter into the airport and once you complete all your security checks and baggage check-ins, then you will go to the mentioned gate where your boarding is going to start. When the boarding is happening, the airlines company, they're going to perform one more check. They'll make sure that you are getting onto the right flight, which is traveling to Dubai. For some reason, if you go to some other gate and try to onboard to a different flight, they're going to stop you by saying that you don't have enough privilege to onboard onto this flight because it is traveling to some other city, whereas you booked a ticket from New York to Dubai. So that airline check is a kind of authorization. Only authorized users are allowed to onboard that flight, which is traveling to Dubai from New York. Very similar style of authentication and authorization we should do inside our backend application as well. Because by using only authentication, we can't build enterprise applications, bigger enterprise applications that are going to have complex security requirements. And those security requirements we can only achieve when we implement both authentication and authorization. . I hope you are clear with the differences between Authentication and Authorization.
