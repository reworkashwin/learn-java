The Spring Data JPA project, it supports a property with the name spring.jpa.hybernate.ddl-auto. By mentioning this property inside the application.properties file, we should be able to control on how the database schema is being managed by Hibernate. We know Spring Data JPA framework, it just follows JPA specifications or standards. But behind the scenes, this framework or this project, it is going to use one of the implementation framework of the JPA specifications. And by default, Spring Data JPA uses Hibernate, which is an ORM framework as the implementation framework behind the scenes. By using this property and assigning a value to this property, we are going to tell to Hibernate on how it has to auto-manage the database schema like tables, columns, constraints, every time our application is starting. Depending on the value that we are providing to this property, the Hibernate can create or update or validate or delete our database schema. So this property, it is going to work in various modes like builder mode, cleaner mode, upgrade mode, inspector mode, and hands-off mode. These are my own definitions. Based upon the value that we are trying to provide to this property, the Hibernate, it is going to enter into any of these mode. As part of the builder mode, the Hibernate is going to create the required tables and database schema. Cleaner mode means it is going to perform the delete operation. Next as part of upgrade mode, the Hibernate, it is going to perform the update of the database tables and schema. Next as part of the inspector mode, the Hibernate framework, it is going to perform the validations during the startup of our application. It is going to validate that our database schema and the entity classes are matching with each other. Otherwise it is going to throw an error. At last we have hands-off mode. As part of this hands-off mode, the Hibernate is not going to do anything. That means we can completely disable the management of the database schema by the Hibernate. In that case, the developers, they are going to take care of database schema and entities manually. So let's try to understand what are the various values that we can provide to this property. The first value that we can assign is create. When we are mentioning these create value during the startup of our application, every time the Hibernate, it is going to drop the existing tables and create the brand new ones. That means all your data, tables, everything, they are going to be created. It's like breaking the old house and building it fresh every time. You can use this value in the scenarios where you are doing early development, testing a new schema. But please note that when you are using this value, all your existing data is going to be lost whenever you restart your application, because the tables are going to be recreated. The next value that these property supports is create-drop. This is very similar to the above one, but when we are using these, apart from creating the tables on startup, the Hibernate, it is also going to drop them when we are shutting down the application. It's like setting up a tent for the day and removing it when our session ends. You need to use these value, especially around the use cases of unit testing and in-memory databases. The third valid value that we can provide is create-only. So when we are mentioning this value, the tables will be created only once. They will never be dropped. It is like build it once and walk away, never tear anything down. There is a clear difference between this create and create-only. When you are using this create, the database tables, they are going to be recreated every time you restart. Whereas with this value, only the first time the database tables will be created. And after that, if you try to restart your application multiple times, then the tables are not going to be recreated because they already created during the very first startup. You can use this value under the use cases of initial migrations. Next we have drop-value. When we are mentioning this value, the Hibernate, it is only going to drop all the tables. It is never going to recreate them. It will be the responsibility of the developer to create the database tables manually. But the dropping of the tables will be taken care by the Hibernate. It is like leveling up the entire building, but not constructing anything new. So the use cases for this will be very rare, mostly used as part of the cleanup by using tools or scripts. Next we have none. When we are using this none, the Hibernate is not going to do anything. The database admin or the developer will be the boss. They have to create the schema they have to manage it and drop it based upon their requirements. And this is the value that is recommended for production environment. If you are building a production ready enterprise application, we typically don't want to give the responsibility of managing the database schema to the Hibernate framework. Typically inside the enterprise applications, what is going to happen is there'll be a separate DBA team who always manages the database schema. Even the developers also, they will not have privilege to create the tables or columns or updating the schema as well. Any change that a developer requires, they have to formally submit a request to the database team after doing enough brainstorming and after receiving enough approvals, the database team, they are going to execute the SQL scripts manually. So once the tables are created, the developers can create the entity classes and repository interfaces. That's what we have followed in the previous section, but I'm trying to discuss this scenario as well so that you can leverage this option, especially during the unit testing scenario or when you are building a low priority application or some demo applications. After the none, we have truncate. When we are using these truncate value, it is going to delete all the rows, but keeps the table structure. That means every time you restart your application, you're going to get empty tables. You need to use this value under the use cases where you want to reset the test data without dropping the tables. Next we have update value. Whenever we are using this value, what the Hibernate framework it is going to do is, it is going to verify what configurations you have defined in your entity class. Accordingly, it will try to match that database schema as well. If there are some mismatches or if you're not following some standards, the framework is going to take care of updating the schema behind the scenes. For example, inside the entity class, you have created a field representing the age column, but this age column is missing inside the corresponding database table. So in this scenario, what is going to happen is during the startup of my application, since the framework, it is going to identify a mismatch between the entity and the database table. It is going to update the existing table to add the new columns. And even if there is a scenario where the entire table is missing, in such scenarios, a brand new table also will be created by the framework. You need to use this value only during the development stages. Like I said, during the production stages, please make sure you are using the none value. This update value is typically not safe for production because it can break schema or cause silent issues behind the scenes. At last, we have validate value. This validate value is like a read-only mode of update. With the update, what is going to happen it is going to identify the issues and automatically update that database schema to fix the mismatches. Whereas with validate, the Hibernate framework, it is going to throw the errors during the startup along with the validation error details so that the developer can decide to update the entity class or the database schema. So when we are using this validate, no changes will be applied onto the database schema. Only validations report will be published during the startup. If there are some validation errors identified, the startup of the application also will fail. This is like someone visiting our house to identify the issues, but they can't touch anything or repair any issue. Whereas the above one, it is like fixing all the small things in our house as needed. First, they are going to do inspection followed by applying the fixes as well. We can use this validate for staging environment and production environment as well. Inside the next slide, I highlighted what are the recommended usages based upon the environment. Inside the local development environment, you can use either update, create, create to drop. During the testing environment, you can use create to drop or validate, whereas inside the production, please make sure you are using either none or validate. By default, when you are using embedded database like H2 database, this property, it is going to have a default value of create-drop. That's why when we are using the H2 database with the internal memory, the tables are created during the startup and they are deleted during the shutdown. Whereas in the scenarios where you are using a normal production database like MySQL, then this property, it is going to default to none value because none is recommended for production applications. That's why to be on safe side, the framework team, they defaulted to none for typical production databases like MySQL, Oracle or Postgres. I hope you are clear on the purpose of these property.
