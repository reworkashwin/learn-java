The Apple company every year, they will try to release new versions of their products like iPhone and Mac. Do you know the reason? Since Apple system want to regularly update their products, they will make sure to adopt all the latest trends and release the new products on a regular basis. Otherwise they can't survive in the business and they may fall behind the competition. In the similar lines, when we are building REST APIs, over time, developers might need to make some changes inside these APIs. These changes include adding new fields, changing the request and response format, modifying the business logic, deprecating any old features. When the developers are making these critical major enhancements in their REST APIs, there is a good chance that issues may arise on the client side who are consuming the REST APIs. Because the developer is trying to change the behavior of the REST API all of a sudden. To avoid breaking existing clients who are consuming the REST APIs, we need to use a standard called API versioning. There are multiple approaches to version our REST APIs, but inside this course, I'm going to explain you four different popular options. Before we try to discuss the details around these options, first, let me explain you by taking a real scenario on why API Versioning is mandatory? Let's imagine there is a food delivery company with the name FoodRush, and inside this company, developers developed many REST APIs that powers mobile applications, website, restaurant partners, delivery partners, and third-party integrations. The mobile apps are being used by 1 million users. Similarly, website is being used by 500k daily visitors. There are 5000 different restaurant partners, 10,000 delivery drivers, and 50 business partners. All these client applications, they are consuming the REST APIs that are exposed by the FoodRush. Let's imagine these developers, they are not following any API versioning. Initially, when they started this FoodRush company, they exposed one of the REST API, which is going to return the restaurant details. So this is the complete API path. Based upon the restaurant_id inside the API path, the REST API, it is going to return the restaurant details by using this response format. This is a JSON object. Inside the JSON object, we have a restaurant key with the restaurant name value, followed by there is an items object. This items is an array object. We can have any number of item objects. Just to save some space, I have provided only one item details. So this is the item ID, item name, item price, and a boolean value indicating whether this item is available right now in this restaurant. So this is the original response. But after a few months or after a few years, since this FoodRush is becoming very popular, the developers on a disaster day, they decided to improve their API by making the response more detailed. This time, if you see inside the response, they made the restaurant as an object. Previously, it used to be a String. Right now, it is an object. So inside this restaurant object, we have various fields like name, ID, and rating. And the items field is renamed to menu items. This is the new name and this is again an array. Inside this array, we can send any number of menu item object details. If you try to understand the item object as well, there are new changes introduced. The pricing is no more a normal String. It is an again an object where they are trying to mention the amount and the currency coming to the availability this is also became an object inside the availability, they have booleans like in-stock and what is the preparation time to prepare an item. Developers they tested their business logic and everything is working and they push this code into the production. And with that, do you know what is going to happen? All the client applications who are consuming these REST API, they are going to get some runtime exceptions, null pointer exceptions, whether it can be inside a website or inside a mobile application or whether inside a restaurant partner application, all of them, they will not work because there are some good amount of critical changes on how the response is being sent. Off course, in real enterprise applications, the developers, they're not going to push these changes all of a sudden, they are going to send an email to all their client applications to update their code. Even if that communication happens as well, still there will be some challenges. Let's imagine these Food Rush developers, they are pushing these new REST API changes on the date, December 20 and the same date at the same time, all the other client applications, they should also make their changes and push into the production. Do you think it is going to happen? Off course not. Few of the client applications, they may say they don't have bandwidth and with that they may not push the changes on December 20. That's why to avoid all these kinds of challenges, we need to follow the API versioning strategy. Now this time, let's try to understand how the same problem will get resolved by using the API versioning. When the food rush company started their business, they are going to build a REST API with this path. Inside the path, if you see, there is a path variable with the value V1. That means any client application who are trying to invoke this API, they are going to receive the original response format. Any new enhancements that the developers, they are going to introduce, they are going to achieve by introducing new REST APIs with a new version number inside the path variable. With this now, the food rush backend employers, they are giving options to the client applications saying that whenever you are ready, please update your code to use the V2 versions so that you will receive a detailed response format based upon the version 2 new feature changes. With this kind of setup, let's try to understand what is really is going to happen in real enterprise applications. During the week 1, the version 2 API is going to be launched. During this time, the version 1 still works. There won't be any disruption and the new features will be available under the V2. Email will be sent to all the client developers about the V2 and requesting them to update their code as soon as possible. During the month 1, let's imagine the mobile application team, they have updated their code they published a new app version that uses V2 REST API. In case if any of the Android application user or iOS application user, if they have not updated their app, still the mobile app is going to work with the older V1. Eventually, when the user updates at their own pace, then it is going to start consuming V2. The same applies for Partner Migration. During the month 2, the partners may migrate to V2. This way we're not forcing any deadlines to any client applications. Let's imagine, after giving enough time to all client applications, during the 6th month, a notice can be sent to all the client application developers saying that this V1 right now it is deprecated and it will be removed in another 6 months. With this deprecation notice, the client applications, they'll make sure to update to latest V2, and with that, the most amount of traffic will come to V2. After the 6 months deprecation notice period, the V1 REST API will be sunset and it is going to be turned off with confidence. And by the time all clients might have successfully migrated, there won't be any business disruption as well. I hope you're clear on why API versioning is very important.
