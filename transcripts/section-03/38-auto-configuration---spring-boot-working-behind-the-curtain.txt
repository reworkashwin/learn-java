Behind the scenes, Spring Boot follows certain principles. One of them is Convention over Configuration. This means Spring Boot, it is going to give you some sensible defaults so that developer don't have to manually configure everything. I can give you an example. Let's imagine you are trying to build an Spring Boot web application. In this kind of scenario, we know a typical web application requires a web server and the code has to be deployed at a given port number. Instead of asking the developer to configure the web server or the port number, what Spring Boot it will do is, it will go with the convention what majority of the developers in the industry are using. With that convention, it will go ahead and configure the project to use the Tomcat server. Once a server is configured, it is also going to assume to start and deploy the code at the port 8080. This way, for all the functionality that Spring Boot is going to provide, it will always go with that default assumptions or default configurations by following this Convention over Configuration. At any point of time, if the developer want to overwrite the default configurations, the developer can do that very easily by going into a file called application.properties. So this is an important file inside the Spring Boot applications. Inside this property file, we should be able to overwrite the default configurations by simply mentioning some properties along with their values. I also mentioned some supporting principles that SpringBoot follows. By using these Opinionated Defaults, Spring Boot is going to provide some pre-configured default.
