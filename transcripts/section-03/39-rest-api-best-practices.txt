And the last one is self-explanatory, whenever your REST API is deleting the data or removing the data from the database, you need to use the DELETE HTTP method. The next best practice is around the response status code that we need to return to the client applications once the processing of the request is completed. If a client application invokes a GET method REST API, we are going to fetch the data and send back the same to the client application. In this kind of scenario, the overall response status code we need to send as 200. Inside the industry, there are some standard HTTP status codes, but 200 means we are trying to convey to the client application that the request is processed successfully. And this 200 means OK as a description. For the POST, we need to send the response either 201 or 202. 201 means the data is created in the backend database, or we can also send 202 that means it is accepted and it will get processed at a later point of time. For the PUT and PATCH, once the update is successful, we can return 200 or 204 that means no content. For the DELETE also, we are going to follow the same, either 200 or 204. So whenever you are sending the response status code by using two series, then it is an indication to the client application that the request is processed successfully at the backend side. For the negative scenarios, we have 400 and 500 errors. 400 we need to send as a response when the bad request is received. If you are expecting the user id in a number format, but client application they are sending in the alphanumeric format, we need to do the validation and make sure to not accept that request. Without processing that request, we need to send the 400 bad request as a status response. 401 means if someone is trying to invoke our REST API without any security. 403 we need to send that someone is trying to access our REST API with valid credentials.
