The solution to the CSRF attack is very simple, but it is a tricky one you need to put some effort to understand this. I am there for you to help you and explain. I just need your attention here. To defeat a CSRF attack, the backend application or the web application needs a way to determine if the HTTP request is legitimately generated via the application's user interface. As of now, the CSRF attack is possible because the Netflix backend server, it is not able to differentiate whether the request came from the Netflix UI code or from the other website UI code. When I say UI code, it is a JavaScript code. For non-web applications, CSRF attack is not possible. For example, if you are building a microservice where a given service is being invoked by another backend microservice, then there is no possibility of CSRF attack because there is no browser involved, there is no cookie involved, and most importantly, there is no hacker involved. CSRF attack is only possible for the web applications which are having the web pages that built by using the JavaScript. The best way to defeat the CSRF attack is, by using a CSRF token. A CSRF token is a secure random token that can be used to prevent the CSRF attacks. This token needs to be unique per user session and should be of large random value to make it difficult to guess. Let's take the same Netflix flow one more time. And this time we're going to introduce CSRF token into the flow. Consider the same Friday scenario, you logged in into your Netflix website. This time, the Netflix backend server, it is going to provide a unique CSRF token along with any other cookies that it generates during the login operation. So this CSRF token also can be shared as a cookie, inside your browser tab right now, you're going to have two cookies. The first one is related to the login cookie, which is going to be used by the UI application to send for all the further requests that it is going to make. The other one is a CSRF token, which is saved inside the cookie. So both these cookies, they tagged to the domain Netflix.com. After one hour, you open the evil.com website and you saw this malicious link. You got tempted and you click on that link. This time also, the hacker is going to execute his malicious script by using the JavaScript, is going to invoke the action on the Netflix. And since the request is going to the Netflix.com domain, the browser, it is going to attach both the cookies, the cookie related to the login and the cookie related to the CSRF token. Even though both the cookies are sent as part of the request, still the Netflix backend server, it is going to reject the request with the 403or401 error. I know you have a million dollar question here. How come backend is able to recognize this is a fake request? Because the browser sent automatically both the cookies. Let me explain the same. Whenever CSRF solution is being implemented by the backend server, we know the backend server initially, it is going to generate the CSRF token and attach it as a cookie. From the next request onwards, the backend server, it is going to expect the CSRF token in two different places. The first one is as part of the cookie, and the second one is under the Request Header. In the hacker scenario, the hacker should be able to send the CSRF token as part of the cookie. And in fact, he's not sending the CSRF token value manually inside the cookie. It is the browser behavior that is sending the CSRF token automatically inside the cookie. But to send the same CSRF token value inside the request header, the client application code or the JavaScript code that is sending the request, it has to read the CSRF token value under this cookie. Once it reads the cookie value, then only it can send the same value inside the request header. So the JavaScript code that is deployed under the Netflix.com website, it should be able to read the cookie because the cookie belongs to the same domain, Netflix.com. But the JavaScript code that is deployed under the evil.com website, this JavaScript code, it cannot read this cookie value or the CSRF token value inside the cookie manually. Because this cookie belongs to the another domain. The browser is not going to allow to read the cookies by a JavaScript code, which belongs to another domain. If you take this script logic prepared by the hacker, though this script is sending the action to the Netflix.com domain. But this script is being executed from a domain evil.com because this web page is deployed under the evil.com domain. Whatever JavaScript code that hacker is going to define here, it is not capable of reading the cookie manually. And with that, it can't send the cookie value inside the Request Header. But the CSRF token value as part of the cookie, it is going to reach to the Netflix backend server, because the browser is going to attach the cookie automatically while sending the request to the Netflix.com. Since the hacker website is able to send the CSRF token value only as part of the cookie, but not as part of the Request Header, the backend application can easily detect these and stop the CSRF attack. The default behavior of the Spring security is, it is going to stop all Non-safe HTTP method requests. . What is the meaning of Non-safe HTTP methods? Any HTTP method that is capable of changing or creating or deleting the data on the backend. Using GET() method, we are only going to read the data. So this is a safe operation. The hacker should be able to only read the data, but not update the data. That's why for safer methods, CSRF protection is not going to be enforced by the Spring security. Whereas for the Non-safer methods, the CSRF protection is going to come into picture. That's why we are getting 401 error when we are invoking the register REST API, even though it is a public page.
