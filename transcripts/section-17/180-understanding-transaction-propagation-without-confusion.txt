When using this @Transactional annotation, we can also configure on how the transaction can be propagated. . So there is a propagation property to this we can pass various valid values by using the propagation enum. If I open this, inside this enum, we have the constants like MANDATORY,NESTED,NEVER,NOT_SUPPORTED, REQIED,REQIRES_NEW and SUPPORTS. . As a backend developer, you should definitely have the knowledge around the transaction propagation and what are the options that you have and which one to use based upon a given scenario. That's why inside this lecture, let's try to understand about what is this transaction propagation along with other details. By using the Transaction Propagation, we are conveying what should happen if a transactional method is called when another transaction already exists. Based upon the transaction propagation value that we are trying to configure, accordingly, the behavior is going to change inside the framework. The very first valid value that we can provide to the propagation is required. This is the famous one. And by default, the framework, it is going to be used this one behind the scenes. When we are using the required, it is an indication to the framework, if a transaction already exists, join into it, otherwise you can start a new one. For example, inside the service1 class, there might be a service method with the name M1. And there might be another service class like service2. And inside these, we may have another method with the name M2. Both of these methods, we have the @Transactional annotation. So when the method one is called from the controller layer, a new transaction is going to be created. Now let's imagine from this method one, the M2 method present inside the another service class is getting invoked. Since we already mentioned this @Transactional annotation on top of this M2 as well, this time, the framework is going to look for the propagation configurations that we have done. With the required, what is going to happen is, since there is already a transaction available by the time this M2 method is invoked, the framework will simply reuse the existing transaction and join into it. . Otherwise, if there is no transaction available, it is going to start a new transaction. We can configure this propagation by using the propagation property and to this, we can provide the enum value called REQUIRED. Even if you are mentioning the simple @Transactional annotation behind the scenes, the required is going to be considered as that default value. For example, here, if you see there is a method with the name placeOrder(). And from this method, we are calling other methods like saveOrder(), saveOrderItems(). So using the saveOrder() method, we are going to save the order details, whereas by using the saveOrderItems(), we're going to save the order item details inside the database table. So when the saveOrder() method is getting executed, a transaction anyway will be created. And when the second method is invoked, the existing transaction will be reused. When we are using required, we are telling there should be one transaction and that transaction will be committed when all the methods are successfully executed. If there is a failure in any of these methods, the framework will rollback all the changes that we have done as part of the single transaction. Why this propagation is default? Because this is the behavior that developers want 90% of the time. After required, we have supports. When we are using supports, we are telling to the framework, if a transaction exists, use it. If not, run without a transaction. So whatever method that we are executing with the supports propagation, it is going to work with or without a transaction. It is going to be used commonly for readOnly operations. You may have a question here. How is it possible to execute a database operation without a transaction? As I highlighted here, usually we need a transaction only for the update operations. The whole purpose of transaction is to rollback the changes in the case of exception. Whereas with the readOnly operation, we're not committing any code changes. That's why even without the transaction, the framework should be able to execute your select query. For such readOnly operations, you can use the propagation as supports. Next we have Mandatory propagation. We can configure these by invoking the mandatory constraint. With this Mandatory propagation, we are telling to the framework, a transaction must already exist if not, throw an exception. You can use this kind of propagation in the scenarios where you want to execute some operations as part of another parent operation. For example, the debit operation must be part of the checkout operation. Calling Debit alone should be illegal. In this kind of scenarios, you're going to simply mention mandatory Propagation for debit operation. So that if someone is trying to invoke this Debit operation directly, it is going to throw an exception. Whereas if the same Debit operation is getting invoked after the checkout operation, then the framework is going to process the logic because a transaction is already exist. Next we have REQUIRES_NEW. Using this propagation, we're telling to the framework, pass the current transaction and start a new transaction always. Developers typically, they may want to use these request new propagation in the scenarios where if main flow fails, still they want to execute the later operation. For example, take a scenario where we have a main operation followed by, we're going to make an entry into some audit related table. In this scenario, even if the main flow operation fails, the operation related to audit log should be committed. The same applies for other kinds of scenarios where we may want to maintain login attempts, failure history, and sending notification. So all these operations, typically we want to execute them regardless of main flow. Under such scenarios, you can use these REQUIRES_NEW propagation. Next we have NOT_SUPPORTED. With this propagation, we're telling to the framework, run without any transaction. Even if there is one exist already, suspend it. So there is a clear difference between these not supported and these optional transaction. These optional transaction is going to use that transaction if it is already available, if there is no transaction, then only it is going to execute without a transaction. But in the scenario of not supported, the logic is always going to be executed without any transaction. If there is already an active transaction available, it is going to suspend it and execute the logic without a transaction. This option you need to use in the scenarios where you don't want any rollback support. Typically in the readOnly operations where you are looking for better performance and no locking. But please be very careful. No transaction means no safety net. You should not try to update any data. The next propagation that we have is NEVER. With this propagation, we're telling to the framework, if a transaction exists, throw exception. Don't execute the business logic. Usually we're going to use these NEVER propagation in the scenarios where we are making externalApiCall(). Let's imagine there is a method M1 and inside this method, we are making an externalApiCall(). On top of this method, if you have a @Transactional annotation, what is going to happen? The transaction is going to keep waiting unless you're going to get the response from the externalApiCall(). . This will result into long logs. That's why as a safety measurement in the scenarios where you don't want to run your logic as part of a DB transaction, you can mark them as propagation as NEVER. So if a developer is invoking these externalApiCall() as part of a database transaction, the framework is going to throw an exception and we'll stop the developer right there. At last we have NESTED propagation. By using these NESTED propagation, we're telling to the framework, create a sub transaction inside the main transaction. So here, if you see, there is a method with the name processItems() on top of this method we have mentioned the propagation as NESTED, while processing the very first method like process item one, a sub transaction will be created, which is dedicated to this method. Similarly, another sub transaction will be created for the second method. In the scenarios where the second method is having some runtime exceptions, then only the changes that are applied as part of the second method will be rolled back and the rollback that is happening inside the process item two, it is not going to have any impact on the data that is applied as part of this process item one, because that is completely independent another sub transaction. So this is like having checkpoints in a video game. If you fail and restart the game, the game is going to start from the checkpoint, but not from the beginning. In this slide, I have given a summary of all the transaction propagations. As I said, REQUIRED propagation, it is always going to join the existing transaction. In case if there is no transaction, it is going to create a NEW transaction. This REQUIRED propagation is useful for Default business logics. SUPPORTS will join the existing transaction, but it is not going to create any new transaction. It is used for readOnly scenarios. And coming to the MANFATORY, the existing transaction must exist. It is never going to create a new transaction. You need to use this MANDATORY in the scenarios where you want to apply some strict rules. Request_NEW will suspend the existing transaction and create a new transaction. You need to use the REQUIRES_NEW in the scenarios where you want to perform some auditing or logging of some data into the database. NOT_SUPPORTED will suspend the existing transaction and it is also not going to create any new transaction this means your entire logic is going to be executed without any transaction. And this propagation is useful in the scenarios of reports. When we are trying to generate the reports, we usually read a lot of data from the databases. While executing those reading operations, we don't want to have any transactions. That's where we can use NOT_SUPPORTED. With NEVER, if there is an existing transaction available, an error will be thrown and it is never going to create a new transaction. This propagation you need to use as a safety mechanism when you are making a call to the external API's. NESTED is going to create the sub transactions as a save points and creating a new condition is always going to be conditional. If you have multiple methods invocation, the new sub transactions will be created based upon multiple method invocations. So with the NESTED, we should be able to do the partial rollbacks. Since required is the most recommended transaction propagation throughout this course, I'm going to use the required only, but if you have any business use cases, which requires different transaction propagation, feel free to use any of the propagations that we have discussed.
