In a typical real enterprise application, hundreds of users will be changing the data inside the database at the same time. If they interfere with each other, data may get messy. That is where isolation levels come into picture. By using these isolation levels, we are going to define walls between the user transactions, but there are some trade-offs for these isolation levels. If you're going with the Low isolation, you may get faster performance, but higher risk of weird data. Whereas with the High Isolation, you're going to get perfect data accuracy, but there'll be slower performance because to provide you the perfect data accuracy, the database is going to enforce some locking mechanisms, which will result into the slower performance as other transactions will be waiting for the database to give access to a given data. To understand these isolation levels in detail first you need to understand three different jargons around the isolation. The first one is Dirty Read. If the system or the framework is allowing you to read another transaction data without committing that transaction, then dirty reading is going to come into picture. For example, if you're reading data that another person changed, but has not saved or committed the transaction yet, if they undo their change, then whatever data that you read will become fake. That's why this concept is called Dirty Read, but there are certain advantages with dirty read. I'll come to that. Next we have Non-Repeatable Read. Let's imagine you read a row and the same rows is updated by someone else. After a few seconds, if you read the same row again, this time you're going to get different data, because someone else updated the row that you are interested. So it is like book changing its text while you are still reading it. The names non-repeatable read is very suitable this scenario because whenever you are repeating the read operation, you may get different data if someone else is updating the same data. If you want to have the same data, you should not repeat the reading operation. That's why this name Non-Repeatable Read. Next we have Phantom Read. Let's imagine of a scenario, you're trying to read 10 different record items from a database table, after you read the records, someone else inserted a new record. And with that, if you try to read and count again, the number of record items, suddenly you are getting 11 rows or 11 items. What happened here? If phantom appeared, what is the meaning of phantom? Phantom means ghost. Suddenly it goes to record appear that's why this name has been given to this concept. I hope you are clear with these three jargons or concept by using these three concepts only I'm going to explain the isolation levels. The very first level of isolation level supported by the Spring Data JPA is, READ_COMMITTED. . If you are defining this isolation level by using the isolation property against to the @Transactional annotation, what you're telling to the framework is, allow a transaction to see the changes the moment they are updated inside another transaction. Even if the other transaction is not committed its data, the vulnerability of this isolation is, we will see Dirty Reads. What are Dirty Reads? We already discussed here, as part of the dirty reads we should be able to read other transaction data even before it's committed to the database. In case if there is some rollback is happening, or if the other transaction undo its change, then whatever data that we read is going to become fake. The same problem is going to happen with this isolation. But you may have a question then under which scenarios we need to use this isolation? Usually this isolation will be used very rarely, but if speed or performance is matter, and if you're looking for high performance and accuracy does not matter under such scenarios, you can use these isolation level. For example, usually inside social media applications like Facebook, they may want to display the likes counter on a post where being off by one is not a disaster. So their accuracy does not matter, but they want to have speed. After these READ_UNCOMMITTED, we have READ_COMMITTED. So this is like next level of isolation. When we are using these READ_COMMITTED, we are telling to the framework, please allow a transaction to read that data only if it is committed by the other transaction. If there is uncommitted data, such data will not be appeared to other transactions. The transactions will only see the data that has been officially committed or saved into the actual database table. So the benefit of this approach is, there won't be any Dirty Reads, but the catch is with this approach, we might still see Non-Repeatable Reads. What is Non-Repeatable Reads? We read a row and someone else updated it and committed it. And if you read it again, and this time we are going to get different data. So with the read committed, we are going to have the non-repeatable reads, but we're not going to have any Dirty Reads. This READ_COMMITTED is a default transaction isolation provided by the databases like Postgres SQL and SQL server. In case, if you want to avoid both these Dirty Read and Non-Repeatable Reads, then you need to go for the isolation with the property with the constant REPEATABLE_READ. With this isolation, what is going to happen is, whenever a transaction read a row, the database freezes that row data for that specific transaction. If some other transaction updated it, our transaction still keeps seeing the version it started with. With that, the benefits are there won't be any Dirty Reads and No Non-Repeatable Reads as well. The catch here is we might still see some Phantom Reads. For example, if new rows are appearing, we can't avoid that. This isolation level is recommended for the scenarios where you want the data to stay consistent while you are executing your transaction, especially in the scenarios where you are generating the financial reports and executing some calculations. So during these calculations, you want the numbers to stay consistent for such scenarios this isolation level is going to be super app. MySQL database is going to follow this isolation level by default. That means whenever you are reading a record or a set of records as part of a transaction, even if some other transactions modify some data in those records under your own transaction, you will see the same initial version data that you started with. So that's a beauty of MySQL database. Yeah, you may have a question? How does the database server is going to maintain my own version, which is specific to my transaction? To achieve all this functionality behind the scenes, the database, they are going to use a concept to call Snapshot. When you are starting a transaction with a isolation level as REPEATABLR_READ behind the scenes the database, it is going to take a mental picture of the data as it exists at that exact millisecond. So for the rest of your transaction, you're going to look at the same picture, whereas the reality is other users, they can still change the live data, they can update rows and delete them. With this, if you query the same rows 10 times in the same transaction, you will see the same exact values every single time, even if another user changed and saved that row in the meantime, as part of the other transaction. How this Snapshot works behind the scenes, they are going to use versioning. The database does not actually freeze the original row, instead, it creates versions. Imagine there is a row representing a bank balance. The row ID is 1 and the balance is 100. There is a transaction A started with the Repeatable Read isolation and it reads the balance as $100. Now another transaction B starts and updates the balance to 200 and commits the data as well. The database now technically has two versions of this row in its storage. Older version will have $100 until the transaction A is completed. Whereas the current version is going to have $200 and this current version will be provided for every other transaction. So if the transaction A reads again, the database checks the transactions A's start time and says, Oh, you belong to the $100 era and with that, the transaction A sees $100 again. So if you closely understand the difference between multiple isolation levels, like READ_COMMITTED and REPEATABLE_READ, under the read committed, the database gives you the latest saved version at the moment you ask. If transaction B saves a change midway through your work, your second read will show the new value. Whereas with the repeatable read, the database gives you the version that was saved at the moment your transaction began. The updated values are always ignored. While these repeatable read freezes the existing rows that you have looked, but it does not freeze the entire table or it is not going to stop in adding another new records. So this results into a weakness called phantoms. If you want to avoid even phantom reads as well, then you need to use the isolation level SERIALIZABLE. So with this isolation level, we are telling to the framework, when a transaction is reading a given set of data, please make sure that no one is updating that data and also no one is introducing a new record under that data. So with that, the benefit is we are always going to have perfect data with zero nightmares. The catch here is it is going to be very slow because the other transactions, they have to wait for each other until the lock is released by the database server on a given data set. So when the lock is going to be released, once your own transaction is done processing, you need to use this SERIALIZABLE isolation under the scenarios where you are executing extremely critical tasks where a hundred percent accuracy is required, like a complex banking transfer involving multiple accounts. To achieve these kinds of isolation, the database often places range logs on the data. For example, if your query is looking for the users between the age 20 and 30, the database is going to log the entire range so that no any other transaction can add a new user until your transaction is done processing. As I said before, this isolation level is useful in the scenarios where you are executing some extremely critical tasks. For example, you're trying to check the balance across five different accounts and ensure the sum is exactly $1 million. If yes, you want to distribute a 1% bonus to all five accounts and verify the sum is now exactly $1,010,000. In this kind of scenario, if another transaction adds a new account or move money between these accounts while you are calculating, your math will be wrong. That's why using transaction with the isolation serializable ensures that for those few milliseconds, the database will stop so that your math calculation stays perfect.
