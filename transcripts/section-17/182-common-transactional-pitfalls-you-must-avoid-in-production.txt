While using @Transactional annotation, there are some common pitfalls that you need to be aware as a backend Java developer. The first pitfall is, if you are calling a @Transactional annotated method from another method inside the same class, the proxy is going to be bypassed and the transaction won't start. For example, here there is a service class with the name UserService. Inside this class, I have this method which is annotated with the @Transactional. In the similar lines, I also have another method with the name setupProfile(). On top of this method also, I am using @Transactional annotation. With this scenario, from the createUser method(), I am trying to invoke this setupProfile() method by using this operator because this method belongs to the same class. In this kind of scenarios, whatever @Transactional annotation that you have mentioned on top of this method, it is not going to have any effect. The reason is, behind the scenes, whenever we are calling a transactional method from within the same class, it is going to skip the proxy entirely no transaction is going to start. Why this behavior? To implement all this transactional related logic behind the scenes, Spring wraps our bean in a proxy object so that it can intercept each called and manage or implement transaction related logic. With the internal calls, you are going to invoke the methods directly, which will bypass the proxy object that the Spring has been created. That's why never ever try to call the methods inside the same service class from another method. Normal methods which does not have any database interaction related logic, such methods you can call them because they don't require any transaction. But if there is a method which is having some data operation related logic, then that is not recommended . instead you need to move that into some other bean and invoke that method. Otherwise, you can club both the methods logic into the same method in the service layer. The next pitfall is around the Non-Public methods. If you are trying to use these @Transactional annotation on top of the private, protect, final or static methods, then this annotation is going to be completely ignored by the framework. Why? because, as I said before Spring framework, it is going to create some proxy objects where it will overwrite the logic that we have provided by appending some transactional related logic. But with the private, protected or final static methods, the proxy cannot overwrite them properly Because in order to overwrite them, the JDK proxies, they need interfaces or public methods. Behind the scenes, JDK is going to use a open-source Java library with the name CGLIB. CGLIB means Code Generated Library. So using this library, behind the scenes, the JDK is going to generate some bytecode or manipulate the existing bytecode as well. So since this CGLIB has some restrictions around these private, protected, final, static methods, using this @Transactional annotation on top of this method is not recommended. It is not going to work. So always keep transactional methods public. Though from Spring 5th version, there are some improvements introduced around the protected methods still, I would recommend you to stick to the public methods. The next pitfall is around the Rollback Rules. We discussed clearly about these rollback in the previous lectures. Let me reiterate one more time, by default, rollback is going to happen only for unchecked exceptions. For the checked exceptions, like IOException or SQLException, the rollback is not going to happen instead, the committing of the data will happen. But you can overwrite this behavior by providing this configuration on top of your method. So this is going to rollback for every exception. In case if you want to ignore the rollback, you need to use the noRollback for property. The last pitfall is around the catching exceptions inside the method. Let's imagine inside your service layer method, you are catching the method and swallowing that method as well. That means inside your catch block, you log the error details and you handle some exception logic, but you never throw that error outside your catch block. And with that, what is going to happen? The framework will never know that there is an exception occurred inside your service method. So that's why the rollback will never come into picture. So here I have given an example inside the catch block, the developer simply logging the exception details and not throwing the exception. So that's why there won't be any rollback in this scenario. For a proper rollback inside your catch block, you can log the details, you can handle the exception, but eventually you need to throw some kind of exception. It can be the same exception or it can be a different exception as well. So please make sure you are aware about all these common pitfalls while you are using @Transactional annotation inside your Spring Boot backend application.
