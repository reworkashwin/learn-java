Please make sure that whenever you are trying to execute a operation that will modify that data first, you need to make sure to use a @Transactional annotation without readOnly inside your service layer followed by you need to mention the @Modifying annotation on top of the repository method that you have defined. This annotation, it also supports a couple of configurations. The first one is flushAutomatically. By default, this flushAutomatically is going to have false value. But if you configure this to true, behind the scenes, what is going to happen is before the framework is trying to execute our own query, it will make sure to commit or flush all that data that it has in its memory into the database. We can read the documentation as well by opening this property. By using this method, we can define whether we should flush the underlying persistent context before executing the modifying query. Just like how Spring Security Framework has security context to store all the security related information . very similarly, the Hibernate framework also has a context with the name persistent context where it is going to store all the snapshots of the entities data, any cache data, everything it is going to store inside these persistent context. So when we are using these ORM frameworks like Hibernate, the developers, what they're going to do instead of executing the SQL statements, they're going to update that data directly inside the entity object. And the developer is going to do this by invoking the setter methods inside the entity object. Let's imagine a developer inside a method is trying to update 10 different columns of a record by invoking the associated setter methods. In this kind of scenario, instead of updating the data, every time a setter method is invoked, what the framework is going to do is, it is going to store the updated entity record object inside the memory, and frequently it will try to flush those memory updates into the database by executing the flush operation. Usually the flush operations will happen whenever the method execution is completed. But with this flushAutomatically, we are telling to the framework, I'm trying to execute a modifying query, please make sure that you are flushing all the underlying persistent context into the database, so that you are modifying query can act upon the latest committed data. Similarly, there is also one more method clearAutomatically(). This property also by default, it is going to have false value, but we can override this to true. With the help of these clearAutomatically(), we're telling to the framework, once you executed my modifying query, please clear the underlying persistent context so that the framework will load that data again from the database, freshly into the persistent context. These are the advanced concept in most of the scenarios. We don't have to use both these properties mentioning the @Modifying annotation should be enough. But if you have scenarios where you want to flush all the existing data into the database, please make sure to use this property by providing the value as true. In the similar lines, if you want to refresh your persistent context data, you need to configure this clearAutomatically() with a true value.
