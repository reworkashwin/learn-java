I said before, when we are using these readOnly as true, along with the @Transactional annotation, it is going to boost the performance of a backend application. Let's try to understand in detail how the performance is going to be improved. Whenever we are setting these readOnly as true, we're telling to the frameworks like Spring Data JPA and Hibernate saying that, Hey, I'm just reading and I'm not performing any updates. With that, the framework is going to skip a bunch of background works. Let's try to understand what are these background activities. Hibernate will skip Dirty checking entirely when we're using readOnly as true. Normally, whenever we are trying to read that data from the database, the Hibernate, it is going to take a snapshot of the every loaded entity to compare later for changes. Whereas with the readOnly true, the entities, they are loaded in readOnly mode. No snapshots stored, no dirty checks during the flush operation or during the commit operation. By using these flush and commit operations, the Hibernate framework, it is going to push the updated changes into the actual database table. When we're using the readOnly true, since Hibernate is not going to create any snapshots, it will result into faster CPU usage, especially if we are loading hundreds or thousands of entities. No iterating over these entities as well to compare the current state with the snapshot state. Let me explain on why Hibernate will try to create a snapshot of the entity object whenever we are loading the data from the database. So if you go to the method that we have created, which is closeContactMessage() here, first we're trying to invoke the findById() method. Since we have mentioned this @Transactional annotation without readOnly configuration, what Hibernate it is going to do, it is going to store the current state of this contact object at line number 84 into a snapshot. And that contact object, which is stored in a snapshot, will have the status value as new. Later, by using the same contact object, I executed the setStatus() method, which is going to update the status from open to closed. And when I'm invoking the save method, the Hibernate first, it will identify whether it has to perform the insert or update operation. Since the contact object that we are providing is having a primary key ID value, the framework will decide to perform the update operation. And during the update operation, the framework has to understand which are the fields that are updated by the business logic. To identify those differences, the framework will try to compare with the snapshot object that is created at the line number 84 with the current contact object that we provided at the line number 89. Unless this snapshot mechanism is present inside the Hibernate framework, it cannot really do the magic. Here taking of the snapshot is mandatory. But what about the scenarios where we are simply performing the read operation? For example, inside this method, we are only going to perform the reading of the data. That's why on top of this method, we have not mentioned @Transactional annotation. Instead, it is going to inherit the readOnly configurations that we have provided on top of the class. With the readOnly configuration, the Hibernate is not going to create any snapshots and that is going to provide some performance improvements. The readOnly configuration, it is going to reduce memory usage, because whatever snapshots that the Hibernate framework used to create for thousands of entity objects, all of them, they are going to eat a surprising amount of heap space, especially when you are executing bigger queries. With the readOnly mode, we are telling to the framework, discard all the loaded snapshots completely, which will result into lower memory footprint per transaction. And this is a great advantage for high concurrency backend applications. With readOnly 2, the framework, it is also going to optimize the flushing operation. Whenever we are setting this readOnly as 2, the Hibernate will set this flush mode to manual there won't be any automatic flushing, because there is nothing to flush for the Hibernate framework. If we are only performing the read operation, since there are no updates are happening to the data, by disabling this automatic flushing, the Hibernate framework is going to bring some performance improvements. The next advantage is around that Database Level Hints. Whenever we are using readOnly configurations, the Spring Data JPA framework, it is going to pass on those readOnly configurations to the underlying JDBC library as well. So databases like PostgreSQL, MySQL, or Oracle, they all support these readOnly configurations. By accepting these configurations, these actual database servers, they can do some performance improvements as well. For example, they are not going to lock any of the rows. The reason is very simple. There is no need of locking of the row because there are no updates are happening. Only reading of the data is happening. The database servers, they can avoid unnecessary transaction overhead, and sometimes they can also reject accidental writes as well when we are passing these readOnly configurations. But please note, not all databases optimizes equally based upon these readOnly configurations. Here I mentioned a scenario as well. Imagine you are trying to fetch 10,000 entities or 10,000 records from the database. Without readOnly, Hibernate will store 10,000 snapshots, which will result into heavy memory and slow flush checks. Whereas with the readOnly, there won't be any snapshots, no checks, quicker executions, and less garbage collector pressure. Always please make sure to mention these readOnly configuration at the service class level, which will be default for all the methods, then override with the plain @Transactional annotation for the right methods. The same best practice we also adopted inside our backend application. So these readOnly configurations, they are going to shine, especially in the Read-heavy scenarios or when loading large object graphs. For smaller queries, the gain is very small, which you can't observe inside your application. But mentioning these readOnly configurations, they are going to future-proof your code, so that in future, if a new developer is trying to build a new method inside your service layer that is going to read thousands of records, under such scenarios, these default configurations will be inherited by the method. Even if the developer forgets about mentioning those readOnly as true, still your application is going to be benefited because you have mentioned this annotation on top of the service layer class.
