Environment variables is a popular approach for external configurations because they work across all operating systems. Java library also they provide a flexibility to the developers to read the environment variables by using system.getenv() method. But when we are using spring boot, we don't have to rely on this code. We can directly inject the environment bean into the business code. By using that environment, we can invoke the getProperty() method and to that method we can pass a property key name. We all know that story. But when you are trying to inject a property along with its value by using environment variable option, then you need to follow a syntax. The syntax is inside your property key, all the letters, they need to be changed to uppercase and replace the dots(.) and hyphens or dashes(-) with the underscores(_). For example, we know spring boot expects a property with the name spring.jpa.show-sql to control the logging of the SQL statements. If you are looking to inject a value against to this property by using environment variable option, then this is how you need to create the environment variable. So all the letters has to be converted into uppercase and the dots has to be replaced with the underscore and any hyphen also should be replaced with underscore. Here you may have a question, why can't I inject directly with this format? We can't use this format because all the operating systems, they expect the environment variables in this format. When we are providing a property key by using these format behind the scenes, the spring boot it is going to understand that this fellow is trying to inject the property value by using environment variable approach. And with that behind the scenes, it is going to convert that environment variable into a spring boot property by converting into a format, which is spring boot can understand. So this automatic conversion is called relaxed binding inside the spring boot. These are the syntaxes that operations team members, they can use to inject a property and its value by using environment variable option. Usually the operations team members, whenever they want to deploy an application or whenever they want to start a backend application, they're going to run a command java-jar and to this command, they're going to provide the location where the backend jar is present. If they simply run this command, then the backend application, it is going to start by using the properties that we have defined inside the application.properties file. Since the operations team member, they're looking for an option to change the property values from outside of the code before running these commands they have to first mention env colon, and what is the environment variable name followed by the value that they want to inject. And towards the end, they need to mention semicolon in case if they want to mention multiple environment variables, they can again repeat these key name and value followed by semicolon, and towards the end, they need to use this command. So this is the syntax inside windows servers. In case the server is using Linux based operating system, then operations team members, they can use these command and syntax. The next popular approach to inject the property values is by using JVM system properties. Whenever we want to pass JVM properties, we need to use a syntax, which is first we need to mention hyphen D followed by what is a property key name along with the value. This time also the operations team members while using Java hyphen jar command, just after Java, they have to use these hyphen D followed by what is the property key name along with its value. This time the operations team member, they don't have to follow the environment variable syntax. They can directly mention the property key name, whatever the Spring Boot is expecting. There is no need of converting to uppercase letter and replacing the dot with underscore and so on. The next common approach is, by using command line arguments. While running these Java command to deploy the application, the operations team members, they can try to inject the property along with its value by using this syntax. First they need to mention two hyphens followed by what is a property key name followed by is equal to and the value of the property, for command line arguments as well we can directly mention the property key name. There is no need of converting into uppercase letters, by using any of these externalizing configurations approach, we should be able to inject a property and its value. When you are injecting a property, there are two possibilities. The first one is there might be no property inside your application.properties file. For example, by using the CLI arguments, if you are trying to inject a property with the key as P1 and value as V1, if there is no already property defined inside the application.properties with the same key, then spring boot framework will try to construct these property and value inside your backend application. The other possibility is, there might be a property with the same key as P1 with a value as V. In this kind of scenario, the spring boot framework will construct these V1 value as these externalizing approaches, they have highest precedence compared to the value that you have defined inside the application.properties file. In case if the same property is being defined with the different values by using all these approaches, then whatever value that you have defined by using command line arguments, it is going to be constructed by the application because command line arguments has highest precedence compared to environment variables and JVM system properties.
