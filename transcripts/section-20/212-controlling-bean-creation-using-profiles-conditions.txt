As of now, inside our Spring Boot backend application, we have created many, many Spring beans by using stereotype annotations like @Component, @Service, @Controller. We also created few Beans by using @Bean annotation as well. But the problem with the Beans that we have created so far is, they are always going to be created. But sometimes we may have a requirement to conditionally create the Beans so that it can open new powerful options to the developers in executing their business logic. This is especially useful when we want to switch configurations, enable features for certain environments, avoiding unnecessary beans inside production. That's why inside this lecture, we are going to focus on understanding on how to conditionally create a Bean. The first and most basic approach is, by using an annotation with the name @ConditionalOnProperty. To this annotation, we can provide the property key and the value by using this name and having value. With this kind of configuration, what is going to happen is whatever bean that you are trying to create inside your method by using this @Bean annotation, this bean will only be created when a given property is present with the configured value. This feature is going to be super useful in the scenarios where you want to create the beans only when the given feature is enabled. We can also control the creation of the bean by using @Profile annotation. Just above the @Bean annotation, we can use @Profile annotation and to this annotation, we can provide the profile name like dev or ! of dev. So based upon the condition that we have given, the bean creation will be controlled by the Spring Boot framework. With this configuration, this data source bean will only be created when the dev profile is activated. We can also create a bean conditionally only if the bean does not already exist. For this, we have to use the annotation @ConditionalOnMissingBean. When we are using this annotation, the bean of this MyService will be created only if there is no bean already created for this class. The next approach is by using @ConditionalOnClass annotation. By using this annotation, we are giving instructions to the framework to create a bean only if a specific class exists in the class path. For example, here I'm telling to create the bean of this MyLibraryBean only if the class of some library is present inside the class path. When I say class paths, I'm referring to the dependent libraries that we use in a typical backend application. There are multiple benefits with the conditional Beans. Developers, they are going to have clean and flexible configurations. We can activate a feature only when needed, reduce memory and startup time, avoid duplicate beans. So whatever approaches that I have discussed so far, I have used the examples by using @Bean annotation, but the same approaches can be used with  stereotype annotations as well, like @Component, @Service. With those annotations as well, we can use these concepts that we have discussed so far. Let me show you a very interesting example where we can use this Conditional Bean Creation concept. Usually in a typical enterprise application, the QA team members, they always going to reach out to the developers saying that to give flexibility to them to log in into the application inside QA environment without any valid password. Because QA team members, they're going to test various scenarios with various username and passwords. Off course usernames, which are emails they can get from that database, but coming to the passwords, they can't remember or they can't make a note of all the plain passwords that they have used while creating an account. With that, they are forced to make a note of all the passwords that they have used during the account creation.
