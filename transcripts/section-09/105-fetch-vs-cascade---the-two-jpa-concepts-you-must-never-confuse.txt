We discussed on a high level about the purpose of fetch type configurations. By using these fetch type configurations, we are going to tell to the Spring Data JPA what needs to happen when related entities are loaded from the database. For example, should the related data be loaded immediately when the parent is being loaded or should it be loaded later only when needed. We can configure either fetch type as lazy or eager. With the eager configurations, the Spring Data JPA framework, it is going to load the related entities as well right away along with the parent entity. In our scenario, when it is loading the company related information, the JPA framework, it is also going to load all the jobs related information as well. If you want to avoid that, you need to mention the fetch type as lazy. I have a real life analogy as well. You can think like eager as ordering the main dish along with the side dish at the same time, even if you're not hungry for the side. In this scenario, what is going to happen ? Even if you're not hungry for the side dish, it's still, it is going to come because you have ordered the main dish and side dish together. Whereas with the lazy, it's like ordering only the main dish first so that at later point of time, if you feel hungry, you are going to order for the side dish. These fetch type configurations we can provide for all these annotations. So all these annotations, they support a parameter with the name fetch to which we can either configure eager or lazy. In case if you're not configuring these fetch type configurations, behind the scenes, the Spring Data JPA framework, it is going to consider some default fetch type based upon the relationship. So for the @OneToOne and @ManyToOne, the default fetch type is eager, whereas for @OneToMany and @ManyToMany, the default fetch type will be lazy. There is a logical reason why Spring team, they have gone with eager for these two relationships whereas lazy for these two configurations, if you see for @OneToOne and @ManyToOne scenario, the child entities are the child objects there will be only one record for the parent record. Since there will be always only one child entity, there won't be any performance problem in loading that single record into the memory. That's why the framework team, they configured eager in this scenario. But let's imagine in the scenario of @OneToMany or @ManyToMany, the child records will be many. For example, for a single company, we may have thousand job records. Loading all those thousand job records eagerly will attract some performance problem. That's why here by default, the framework is going to consider lazy in case if the developer have a scenario where the developer is always going to look for the child records, then the developer always have an option to override these to the eager. I hope these default fetch type makes sense to you. You can remember these default fetch type very easily. If you are seeing one towards the end, you can assume eager because loading one record won't create any problem. Whereas if you're seeing many towards the end, the default fetch type will be lazy. Next we have cascade configurations as well. So what is cascade? By using cascade configurations, we are telling to the Spring Data JPA framework, if I do something to the parent entity, automatically do it for the child entity as well. This is like a chain reaction. Changes made to one entity get automatically applied to the related entities. For example, if I am deleting a company record, all the associated child records will be deleted automatically by the Spring Data JPA. So why we need to configure these cascade configurations? To reduce the boilerplate code, to avoid manually saving or deleting the related entities, to keep the entities always in sync automatically. In the code configurations that we have done previously, we configure the cascade configurations with the constant as all. Apart from ALL, we also have other cascade types like PERSIST, MERGE, REMOVE, REFRESH, DETACH. So let's try to understand what is the meaning of each of these cascade type. If we are mentioning the cascade type as PERSIST, then the Spring Data JPA framework is going to save the child entity when the parent is being saved. For any other operations like delete or update, the Spring Data JPA framework is not going to make any modifications onto the child record. The MERGE is like update when you are trying to update the parent entity, automatically the child entity records also will be updated. Remove is for delete. When you are deleting a parent record, all the associated child records will be deleted by the framework. Next, we have REFRESH inside the Hibernate framework, at regular intervals of time, the framework, it is going to refresh the parent object. As part of this refresh operation, the framework is going to fetch the latest data from the database for that record. So when the parent is being refreshed, we are telling to the framework to refresh the child records as well. Next we have DETACH with detach, whenever the parent is being detached from the persistent context, the child entities will also be detached. So what is persistent context ? The Hibernate framework, it is going to store all the entities data, like companies data or jobs data inside a small memory called persistent context. Any changes that we are applying by using the Company object or the Jobs object, those changes will be tracked by the persistent context so that when we are executing methods like save() or remove(), those changes will be applied inside the database. In case if we don't want the framework to track our objects for some scenario, then we have an option of detaching those objects from the persistent context. Under such scenarios, we're telling whenever the parent is being detached, please also detach the child records as well. In case if you want to apply all this cascade type, you also have an option of ALL. If you want to mention only two cascade types, you can do that as well by mentioning persist and remove separately.
