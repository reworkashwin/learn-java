Typically, whenever a Java developer decided to communicate with the database using the Java code, these are the steps that the developer has to follow. All these steps are mandatory. The first one is, the developer has to establish a connection with the database that he's trying to communicate. So this is how the code is going to look like. So by using the DriverManager class available inside the JDBC library, the developer has to invoke the getConnection() method. And to this method, the developer has to provide the database details like what is the URL? What is the username and password? By using all these details, the JDBC library, it is going to give the connection object as a return value. So once the connection is established, the developer has to manually create an SQL statement that he wants to execute on top of the database. For example, here, the developer wants to execute a SQL query, select * from companies. Basically, the developer is trying to fetch all the company's data from the company's table. So to prepare an SQL statement using these connection objects, the developer has to invoke the prepareStatement() method available inside the JDBC library. From these, the JDBC library, it is going to return the PreparedStatement object. Once the PreparedStatement object is created by passing the SQL statement, as a next step, the developer has to execute the query. So to execute the query by using the PreparedStatement object, the developer has to invoke the executeQuery() method, which is going to return all the database records in the form of ResultSet object. So once the ResultSet object is available, the developer has to process the results and convert that database records into Java POJO objects. Because at the end of the day, when we want to send this data to a client application or when we want to execute some business logic inside our backend code, we need to transform that database records into Java objects. Unless we convert that data into Java objects, the Java code cannot understand our data. So that's why the developer has to write a lot of logic to iterate all the results available inside this ResultSet. And during the iteration of the each record, the developer has to look for the exact columns like ID, name of the company, size of the company. So by mentioning all those exact column names like rs.get(), and to this get method, the developer has to pass the column name. To this, the framework is going to return the ID value, and the same ID value needs to be transformed or saved into a Java object, which is representing the company's data table. So that the entire process of processing the results and converting into POJO objects needs to be taken care by the developer. Once the database records are available inside the POJO objects, the developer can continue with the next set of business logic. But before that, the developer has to make sure to close the resources and connections that are opened during the execution of this logic. Here inside this code, by using this logic, we have opened a connection with that database. So by using a finally block, the developer has to make sure to close the connection by invoking the close() method. Once the resources and connections are closed, the developer also should make sure to handle the exceptions properly. Whichever code that is written by using the JDBC library, this code, it is going to throw an checked exception with the name SqlException. Since it is a checked exception, the developer either has to catch the exception and handle it or it can be thrown to the parent methods by using the throws keyword. So here the developer is also responsible to handle the exceptions properly. If you look closely at all these steps that we need to follow by using the JDBC library, Most of them are boilerplate code. That means the developer don't have to worry. The frameworks can take care of them. The developer should only pass the information to the framework on what SQL statement has to be executed by the framework onto the database so that the business logic is going to fetch the required data from the database. So creating a connection, creating an SQL statement, executing it, processing the results, using the connections, handling the exceptions, all of them are unnecessary steps from the developer perspective. They are going to consume a lot of time of the developer. That's why frameworks like Spring Data JPA came into picture, which are going to take care of all the steps that we have discussed so far. The developer has to only communicate to the framework on what is the logic or what is the SQL statement or what is the where condition that he want to execute on top of a database table to fetch the data.
