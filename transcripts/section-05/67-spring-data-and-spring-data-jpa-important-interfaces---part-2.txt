Before I try to close this lecture, I want to share some quick tips around the Spring Data and Spring Data JPA important interfaces. There is a @Repository annotation inside the Spring core library. This is a stereotype annotation, which is similar to @Component, @Service and @Controller annotation. Don't try to get confused between this annotation and the Repository interface that we have seen in this lecture. We need to use this annotation on top of a Java class to convey to the Spring framework to create a bean of a given class and treat that as a Repository layer bean. Whereas the Repository interface that we have discussed, we need to use that to develop a Java interface component inside our application, using which we can interact with the database tables. As we saw previously, Spring Data follows interface segregation principle, providing multiple Repository interfaces that extend one other. This allows applications to adopt only the functionality they need, avoiding unnecessary complexity. Many times developers, they are going to have a question. Why Spring Data and Spring Data JPA team, they are providing so many interfaces. Why can't they put everything in a single interface? Since it is a bad practice to put everything, the Spring Data team, they followed interface segregation principle. Since we have many interfaces available, if there is a scenario where I am looking only for the paging and sorting related requirements, then I can only extend the paging and sorting related interface and ignoring CRUD related interface and JPA related interfaces.
