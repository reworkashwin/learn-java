From our back-end REST API, we are trying to send the response to the client application by using the entity class itself. Here, we are trying to send the List of Company records as a response to the client applications. The same we can also confirm by looking at the response under the postman. We receive List of Company objects as a response. Each of these fields inside the JSON object, they represent a database column inside the company's table. But sending these entity objects directly to your client applications is not a good practice. Instead, we should follow DTO pattern. Let me explain what is this pattern and why we should use it. DTO means Data Transfer Object. The DTO pattern is a design pattern that allows us to transfer data between different parts of our application and what are these DTOs? These are the simple objects that contains only data and they will not have any business logic. This makes them ideal for transferring data between different layers of our application such as presentation layer and the data access layer. There are certain advantages or benefits that we are going to get when we transfer the data by using DTO objects. Let's consider a scenario there is a Customer entity class representing customers table inside my database. And this entity class, it is going to have various fields like name, email, mobileNumber, password, and SSN. All these java fields, they represent database columns under the customers database table. If I try to share the customers data by using this Customer entity objects to other layers like controller layer or presentation layer or to the client applications, there will be a problem. We are going to share some details unnecessarily to other layers. For example, this password and SSN, I don't want to share them to other layers or to the client applications because these fields, they are going to hold sensitive information. In this kind of scenario, what we have to do is we have to create a DTO class. Here, I have created a DTO class with the name CustomerDto. Inside this DTO class, we can define the java fields from the entity class, which we are comfortable sharing with the other layers. Here, I have mentioned the fields like name, email, and mobileNumber, but not password or SSN. This is how we are going to restrict ourselves from sharing unnecessary data to other layers or to the client applications. Once the customer data is loaded from the database and the object of this CustomerEntity is created inside the repository layer, what do we have to do ? We have to execute some mapper logic. By using this mapper logic, we are going to transfer the required fields from entity object to DTO object. For example, I can transfer name, email, and mobileNumber. And during the copying of this data, if required, I can write some business logic to transform that data as well. Let's imagine there is a gender column inside the database and inside the database, we are going to store gender value like M and F. But while you are sharing this information to the client applications, you may want to send the gender information as male and female. So this conversion from M to male and F to female, we can write under this mapper logic so that by the time the gender data is copied from DB entity to DTO class, we're going to have these transformed data. So this is one other advantage. In the similar lines, we have many other advantages. Let me show you one significant advantage. Let's imagine you have two different database tables like Customer and Accounts. Inside the customer table, we are going to have all the customer related personal information. Whereas under the accounts, we're going to have the bank account details of the customer like accountNumber, accountType, and branchAddress. This way we're going to have a good amount of tables in real applications as well. Let's imagine there is a scenario where one of the client application, it can be an UI application or it can be an another backend API. They want the information both from these Customer and Accounts table. If you're not following these DTO pattern, you can send either customer or accounts data as an output from your REST API. You can't send two different objects as an output. And with that, what is going to happen? The client applications, they have to make two different REST API calls, one to fetch the customer information and the other one to fetch the accounts information. And this is going to introduce unnecessary traffic into your web application. To handle these kinds of scenarios also, we can use DTO pattern. I'm going to create a DTO class with the name like CustomerDetailsDto. Inside these DTO class, I can define all the Java fields from both these entities. So I want to send name, email, mobileNumber. Similarly, I want to send these three fields as well. Any field which I don't want to send from these entities, I can ignore it. And I'm not going to maintain the corresponding Java field inside the DTO. So once I have the DTO class, I need to write the mapper logic. This mapper logic will be executed once the customer and accounts details are loaded from the database. Inside this mapper logic, I'm going to write a logic to read the required fields from the customer and populate into these three respective fields. In the similar lines, I'm also going to read the required information from the accounts object and copy them into the DTO class. And once the DTO class is ready, we are going to send the response by using the object of this class. This way, the client applications, they're going to receive the combined information with a single REST API call. I hope this is clear. I also highlighted some of the benefits of using the DTO pattern. The first one is around reducing the network traffic. Just now we discussed this scenario. When you are using the DTO pattern, you can batch up multiple pieces of data into a single object, which can reduce the number of network requests that need to be made by the client applications. The next advantage is around the serialization. Within these DTO class or within this mapper logic, we can control on how our objects will be transferred over the network. What is serialization? Serialization is nothing but sending the data inside the Java objects to the client applications by using formats like XML, JSON, and so on. Let's imagine today you are trying to send the information to the client applications by using the JSON format. In the future, if you want to use some other format instead of JSON, then the changes will be only inside the mapper logic and inside these DTO class. There won't be any other changes inside other layers like controller layer or inside the repository layer. This way, you can always control on how the data needs to be transformed to the client applications by using these DTO pattern. And the last advantage is around the decoupling of the layers. When we are using the DTOs, the different layers that we are going to have inside our application, like presentation layer, data access layer, they're going to be decoupled from each other. Even if there is a change is happening inside these entity classes, it is not going to affect the presentation layer or the controller layer, because for the other layers, we're going to transfer the data by using the DTO class, but not using the entity objects.
