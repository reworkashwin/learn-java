That's why we have to invalidate the entire cache by using @CacheEvict annotation. Before closing this lecture, let me share more details on how Spring decides the cache key. Under the case one, I have taken an example where a method has a single input parameter. With this kind of method, if you're trying to use @Cacheable annotation, as I highlighted, whatever method input data that you're trying to provide, that will be considered as a key. For example, if someone is invoking this method by using ID as 1, then the key will be 1. And for this key, whatever method result will be stored. So this can be some user1, and similarly, if someone is providing key as 2, then for that key 2, a different user two will be stored inside the cache. In future, if someone is calling the same method with one of these keys, the data will be fetched from the cache, otherwise a new cache entry will be made based upon the new key value. Coming to the scenario where method has multiple parameters, it can be two or three or four or any number of parameters. So in this scenario, the cache key will be combination of all parameters. For example here, based upon the email and the role that I'm providing behind the scenes, the framework is going to calculate the hash value and that hash value will be treated as a cache key value. Now let's see a scenario where a method has no parameters. So we already discussed about this. If a method is not accepting any input parameter, then behind the scenes, Spring is going to use SimpleKey.Empty as a key. So there'll be only one cache entry and the method executes only once subsequent calls always use the same cache. As I said before, behind the scenes, the Spring framework, it is going to use SimpleKeyGenerator to calculate the cache key. The thumb rule is very simple. 0 params means empty key will be used. 1 param means the param value itself. In the scenario of multiple params, all the param values will be used to calculate the hash value and that hash value will become the cache key value. Now there might be some scenarios where you may want to provide your own custom cache key. Under these scenarios, you need to use the key attribute against to the @Cacheable annotation. For example, here, I'm providing to construct this ID value as a key, which is the default behavior. Even if I don't provide this, anyway, the framework is going to construct this as a cache key. But if you consider a scenario like where an user object is being provided as an input to this method. Under these kinds of scenarios, it is always advisable to define your own custom cache key. So here by using the key property, I'm providing a key value by mentioning this hash user.email. So this is the Spring Expression language. When you are mentioning hash user, we are telling to the framework to refer to these user attribute and insert this user object. We are going to have a email field, that email field we want to consider as a cache key. So these are the other common examples that you can use when you are using key attribute. So why custom key matters? Defining the custom key is a must when your method is accepting objects as an input parameter. So in this scenario, if you're not using key attribute to the @Cacheable annotation, then these user objects may be treated as differently because every user object, they're going to have different hash codes, even though the content inside the user object is same, but still two different user objects, they're going to be considered as different because their hash code is different. That's why in this kind of scenarios, the cache is not going to work as expected. The recommendation approach is, always to use the key attribute to define your own custom key, and by using this key attributes, you can provide any property value of the user object to consider as a cache key. So these are the common mistakes that developers usually do when implementing caching, please try to avoid them. Similarly, don't try to use the entire object as a cache key, and finally, don't forget that same method with different params is equal to different keys.
