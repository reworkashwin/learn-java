Inside this lecture, we're going to focus on understanding on how to handle the passwords in real applications. As a beginner into the software development, any junior developer will assume that the passwords will be stored as it is inside the storage systems like database. For example, if an end user set the password as 12345 during the registration process, you might be thinking that the same value will be stored inside the database so that during the login operation, we can compare the already stored password with the user entered password and if they are same, we can allow the login operation. This can be your assumption. That is what I also predicted inside this slide as well. This is the flow that is going to happen when you save the passwords using plain text inside the database. But this approach is not recommended for production applications or for enterprise applications. The reason is very simple. If you start storing the passwords in a plain text inside a storage system like database, it is going to bring some integrity and confidentiality issues, anyone who has access to your production database, they should be able to query the User table. Inside this user table, they are going to have both username and password. Since the password is stored using plain text, with that they can mimic an end user and perform any actions that they want. That's why this approach is not recommended for production applications. Instead, we need to look for an option which is going to help us to store the password in a format which can't be understood by anyone, even though if they get access to the production database. Inside the industry, when we want to transfer the data from one form to other form, we have three different options with the name Encoding, Encryption, and Hashing. Using all these approaches, we should be able to avoid storing the plain text password inside the database. But not all of these options are going to be suitable for us. We need to discuss one by one and get rid of the options which are not suitable for converting password values. The first option is Encoding. Encoding is defined as a process of converting data from one form to another, and it has nothing to do with the cryptography. When I say cryptography is a special subject that deals with the secret data, secret communication, all such concepts will come under the cryptography. With the help of encoding, we should be able to convert our plain text password like 1 2 3 4 5 into some encoded value. So by looking at this encoded value, no one can understand what is my original password. But since this encoding does not involve any secret, the encoded value can be completely reversible into a plain text value. If someone have my password encoded value, they should be easily reverse the encoding process with a concept called decoding. By giving my encoded value as an input to this decoding process, they can easily derive my plain text password. We have various encoding algorithms inside the industry like ASCII, BASE64, and UNICODE. Regardless of what algorithm you're trying to use to perform the encoding, all of them have a drawback, which is anyone can easily decode your original text value. That's why this option is not suitable to convert our passwords into some other format. With that, we can move on to the Encryption. Encryption is a process of transforming data in such a way that guarantees confidentiality. To achieve this confidentiality, encryption requires the use of a secret, which in cryptographic terms we call it as a key. For an encryption algorithm, we can feed two inputs like 12345 which is the end-user password, followed by some secret we can inject as another input. By using both these values, the encryption algorithms are going to convert the plain text value that you have feed into some other encrypted data. If anyone has this encrypted data, they cannot derive the original text value by using decryption process unless they know the secret key that we have used during the encryption process. As long as someone is able to securely store the secret that they have used during the encryption process, no one should be able to derive the original text value. But in real applications, we can never guarantee that no one can see the secret key. Server admins, operations team members, or platform team members, since they are going to have access to the production servers, they can easily understand what is the secret that you are using inside the code for the encryption. Once they have the secret and encryption algorithm details, they can easily perform the decrypt operation and derive the plain text value. Since there is a risk of losing the secret key, this option also is not suitable to convert the passwords. With that, we left only with the Hashing process. Inside Hashing, we are going to feed our plain text password values to a hashing algorithm. These hashing algorithms are going to convert your plain text password value to a hash value. Plain data that once hashed, it is non-reversible. For example, if I feed a plain text password like 12345 to my hash function, and this hash function is going to derive some hash value for my plain text password value, even if someone gets hold of this hash value, they cannot really reverse the value to the plain text value. Because hashing is always a one-way process and hashing algorithms, they don't support reversing the hash value into a plain text value. This hashing seems to be a good option to convert our passwords before storing them into the database. I know you may have too many questions at this point of time around Hashing. Anyone who wants to decrypt the encrypted text to a plain text value, they need to provide the secret key that is involved during the encryption process. Otherwise, they're not going to get the plain text value. Whereas inside the hashing, when a plain text value is feed to a hash function, the input value will be converted into a hash value or in other words, we can also call it as digest value. Once a hash value is calculated for a plain text value, we cannot reverse the hash to a plain text value. There is no reverse mechanism inside the hashing.
