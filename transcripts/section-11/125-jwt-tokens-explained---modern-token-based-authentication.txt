So this is the beauty of tokens. Inside the industry, whenever someone wants to generate a token, there are two flavors of tokens. Let's try to understand what are those. The first category of token format is Opaque tokens and the second category is JSON Web Tokens in short form, we call them as JWT. Inside industry, people pronounce this JWT as JOT. When someone says JOT tokens, they are referring to these JSON web tokens. There is a very clear difference between these two tokens. Opaque tokens are typically random Strings with no inherent meaning. Whatever token right now Spring Security framework is generating with the help of JSESSIONID cookie, that token value is of Opaque token. It does not have any inherent meaning. It is a randomly generated string value, which is associated for a user session. These type of tokens are used to reference the authentication information stored on the server side. When someone is using Opaque tokens, the client applications, they need to make sure they are sending the same token to the backend server. On the backend application, the framework is going to check whether the token is really generated by itself. If not, it is going to throw an error. Coming to the JSON web tokens or JOT tokens, these tokens are self-contained that consists of three parts. Usually a Jot token will have a header, payload and a signature part. The beauty of JWT tokens is, we can share certain information about the end-user inside this token itself. For example, I can share the email details of the user or the username of the user, mobile number or the roles of the user. So all this kind of user-related information I can embed inside the Jot token itself, so that when we share the same to the client application or to the UI application, they can try to read the information present inside the Jot token and accordingly they can understand what are the end-user details. One more advantage of Jot tokens is, when we are generating a Jot token, we can set an expiration time. I can generate a Jot token with one hour expiration or one day expiration. It's up to me what kind of expiration time that I want to set. Since the expiration time is going to be embedded inside the Jot token, the client applications also they can understand when this token is going to expire. If the token is expired, they can redirect the end-user to the login page or they can make a request to the backend server behind the scenes asking for a new Jot token. Since these Jot tokens have many advantages inside the industry, you will see these tokens are being used extensively compared to the Opaque tokens. Jot is the most common and favorite token type that many systems use these days due to its special features and advantages. Jot tokens, they can be used both in the scenarios of authorization and authentication along with the information exchange, which means you can share certain user-related data in the token itself, which will reduce the burden of maintaining such details in the session or on the client or server side. But if you're trying to share the same JWT token with a web application, which is available inside the web, under such scenarios, you need to make sure you are definitely digitally signing your JWT token with the help of the last part of the token.
