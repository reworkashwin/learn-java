Understanding on how the digital signatures are going to work inside JWT token is somewhat complex. Let's try to put some good efforts in understanding the same. Whenever a back-end decided to generate a signature for a JWT token that it is going to issue, it is going to follow a formula. The formula is, first, it is going to decide which hashing algorithm it is going to use. Let's assume the back-end application is using SHA-256 algorithm. Once the hash algorithm is decided, whatever header content and payload content that we have inside the JWT token, they are going to be encoded with the help of base64 urlencoding. As you can see here, first, they are trying to convert the header into an encoded value. Similarly, the payload also, they are trying to convert the encoded value. In between them, there are going to be a period. This entire string of header encoded value, period, and payload encoded value will be the first input parameter to the hashing algorithm. Along with this first input parameter, the hashing algorithm, it is also going to receive a secret value which is known only to the back-end server or the back-end application. By using both these input parameters, the hashing algorithm, it is going to return a hash value or a digest value. We already discussed about what is hashing. Based upon the hash value, no one can reverse the plain text value. That means no one can see the secret value that is used by the back-end server to generate that digital signature. I hope you are clear with this formula. Now let's try to understand what is going to happen during the login operation. During the login operation, the back-end application, it is going to generate the header content and the body or the payload content of the JWT token. Inside the header, as I said previously, it is going to have some metadata information like what hashing algorithm it has used and what is the type of the token. So the type of the token here, it is mentioned as JWT token. This plain text header value will be encoded with the help of base64. And this is how it is going to look like. The same we are going to share to the UI application as well. In between this header and payload, there will be a period the same applies for payload and signature as well. Now coming to the body payload, inside the body payload, the back-end application can populate all the user-related information based upon their business requirements like username, email, address, mobile number, when is the token is issued, what is the expiration time of the token. Once the payload is generated, the same will be encoded and this is how it is going to look like. So these encoded values like header and body payload, we are going to share to the UI application as well. Since this follows encoding process, the UI applications are the client applications they can simply run the decoding process and they can decode these values into plain text and they can easily understand what is the username, what is the mobile number and what is the expiration time, all such details they can easily understand. Now coming to the signature part, the signature part is going to be generated based upon the formula that we have discussed here. The header encoded value and payload encoded value with a period along with the secret will be used to generate a hash value. The same hash value will be present inside the signature part of the token. So if you see here, we're sharing everything to the client application as part of the Jwt token. We are sharing the header encoded value, payload encoded value along with the signature. This is what is going to happen during the login operation. Now let's try to understand how the validation on the backend is going to happen to identify whether the tampering of the token is done or not. First, let's imagine the happy flow with the end user or the client application they didn't tamper any of the content. They're trying to send whatever Jwt token that they have received for the next REST API invocations. In this scenario, what backend application it is going to do is it will take the encoded header and encoded payload inside the Jwt token, and it is going to run the same formula along with the secret that is known to only itself. Since the encoded header and encoded payload and secret, everything is same, the backend application, it is going to get the same hash value. If the hash values are matching, then the backend application is going to allow the end user to invoke the secure REST API. If the hash values are not matching, that means someone tampered the token. Under such scenarios, the not equal scenario will come into picture and the REST API invocation will be stopped. Let's assume one of the end user or the client application, they tampered the content inside the payload. Maybe instead of John Doe, they changed the name to Madan. Let's try to understand how the backend application can detect this tampering scenario. Since the backend is going to use the same formula to calculate the new hash value. This time, since the payload content change, the hash value is going to be different compared to what it has inside the signature. Since the hash values are not matching, it will end up into the not equal scenario. So changing the signature is not even an option to the client applications or to the end user because to generate a new signature, they need to know the secret.
