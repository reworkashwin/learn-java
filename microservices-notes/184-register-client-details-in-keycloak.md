# Register Client Details in KeyCloak for Client Credentials Flow

## Introduction

We have an auth server running. But it doesn't know about any external applications yet. Before a client can request an access token, it must be **registered** with the auth server. This is like getting a security badge before you can enter a building.

Let's walk through the exact steps to register a client in Keycloak, configured specifically for the Client Credentials grant type flow.

---

## Why Registration is Required

Think about it — should any random application be able to request tokens from your auth server? Absolutely not.

Registration serves as a **vetting process**:
1. The external application's team approaches the Keycloak admin
2. Business leaders approve the access request
3. The admin registers the client with specific permissions
4. The client receives credentials (`client_id` + `client_secret`)

Only after this process can the client request access tokens.

---

## Step-by-Step Client Registration

### Step 1: Navigate to Clients

In the Keycloak admin console:
- Verify you're in the correct **realm** (master)
- Click **Clients** in the left sidebar
- You'll see some default clients that Keycloak creates internally

### Step 2: Create New Client

Click **Create client** and configure:

| Field | Value | Explanation |
|-------|-------|-------------|
| **Client type** | OpenID Connect | Selecting OIDC automatically includes OAuth2 |
| **Client ID** | `easybank-callcenter-CC` | The "username" — CC suffix = Client Credentials |
| **Name** | EasyBank Call Center App | Human-readable display name |
| **Description** | EasyBank Call Center App | Optional description |

Click **Next**.

### Step 3: Configure Authentication and Grant Flow

This is the critical part:

| Setting | Value | Why |
|---------|-------|-----|
| **Client Authentication** | ✅ Enabled | The client must prove its identity with credentials |
| **Standard Flow** | ❌ Disabled | This is for browser-based login (not our case) |
| **Direct Access Grants** | ❌ Disabled | This is for username/password flow (not our case) |
| **Service Account Roles** | ✅ Enabled | This enables Client Credentials grant type flow |

The **Service Account Roles** checkbox is the one that matters. Click the help icon and it confirms: *"Allows you to authenticate this client to Keycloak and retrieve access token dedicated to this client. In terms of OAuth2 specification, this enables support of 'Client Credentials Grant' for this client."*

⚠️ **Common Mistake:** Leaving Standard Flow or Direct Access Grants enabled when you only need Client Credentials. Each flow serves a different scenario — only enable what you need.

### Step 4: Skip Root URL and Save

Leave the **Root URL** and **Home URL** fields empty. Click **Save**.

---

## The Auto-Generated Client Secret

After saving, the client is created. But notice — you never entered a password/secret. Keycloak auto-generates one.

To find it:
1. Click on your newly created client
2. Go to the **Credentials** tab
3. Click the eye icon next to **Client Secret** to reveal it

You now have:
- **Client ID:** `easybank-callcenter-CC`
- **Client Secret:** *(auto-generated by Keycloak)*
- **Client Authenticator:** Client ID and Secret (default method)

Other authenticator options include Signed JWT and X.509 certificates, but Client ID + Secret is the most common approach.

💡 **Pro Tip:** In production, treat the client secret like a password — store it in a secrets manager (HashiCorp Vault, AWS Secrets Manager), never in code or configuration files.

---

## ✅ Key Takeaways

- Clients must be registered with the auth server before they can request tokens
- Select **OpenID Connect** as the client type (it includes OAuth2 automatically)
- Enable **Client Authentication** to require credentials
- Enable **Service Account Roles** to activate Client Credentials grant type flow
- Disable Standard Flow and Direct Access Grants (they're for different scenarios)
- Keycloak auto-generates the client secret — find it in the Credentials tab
- In production, protect client secrets using a secrets manager
